# 3DGS-LM -- ROCm/HIP port plan (lead: linux-gfx90a / MI250X, CDNA2 wave64, ROCm 7.2.1)

## Project
- Name: 3DGS-LM
- Upstream: https://github.com/lukasHoel/3DGS-LM (lukasHoel, ICCV 2025 "3DGS-LM: Faster Gaussian-Splatting Optimization with Levenberg-Marquardt")
- Default branch: main; cloned read-only at projects/3DGS-LM/src (HEAD d6db64b, "Add files via upload" -- a flat upload, no submodule git; diff-gaussian-rasterization and simple-knn are vendored in-tree).
- What it is: the graphdeco-inria 3DGS trainer with the ADAM optimizer replaced by a Gauss-Newton / Levenberg-Marquardt step. The rasterizer submodule is a MODIFIED `diff-gaussian-rasterization` that adds the LM Jacobian-vector-product CUDA kernels (the distinguishing piece). The LM normal-equations solve is matrix-free PCG in pure PyTorch.

## Existing AMD support
None. No HIP path, no `USE_ROCM`/`__HIP__` guards, no makefile.hip, no ROCm fork or PR upstream. Pure CUDA (CUDAExtension + raw `__shfl_*`/`cub::Warp*` + `cudaMallocManaged`). Decision: a fresh CUDA->HIP port via Strategy B is warranted and adds clear value (no OpenCL/Vulkan alternative exists either). This is NOT a perf-rewrite candidate in the CUTLASS/wgmma sense (no CUTLASS/CuTe/MMA/tensor-core code anywhere -- it is hand-written splatting + JVP kernels), so a correctness-first mechanical port is the right and complete strategy; no AMD-native (CK/rocWMMA) rewrite is wanted.

## Build classification: torch-extension (Strategy B)
Evidence: two PyTorch `CUDAExtension` modules, no standalone CMake build is used (the CMakeLists.txt under diff-gaussian-rasterization/third_party/glm is GLM's own; the rasterizer ships a top-level CMakeLists.txt but the build path is setup.py).
- `submodules/diff-gaussian-rasterization/setup.py`: `CUDAExtension("diff_gaussian_rasterization._C", sources=[rasterizer_impl.cu, forward.cu, backward.cu, gsgn.cu, rasterize_points.cu, ext.cpp], extra_compile_args={"nvcc":[ "-I.../third_party/glm/", "--generate-line-info", "-lineinfo", "--use_fast_math"]})` + `BuildExtension`.
- `submodules/simple-knn/setup.py`: `CUDAExtension("simple_knn._C", sources=[spatial.cu, simple_knn.cu, ext.cpp])` + `BuildExtension`.
- `ext.cpp` uses `PYBIND11_MODULE(TORCH_EXTENSION_NAME, ...)` + `py::class_<GSGNDataSpec>` (relevant only to the gfx1151 follower; see delta note).
ext_type set to `torch-extension` in upstream.json and status.json.

## Port strategy: Strategy B (torch auto-hipify)
Torch's ROCm `CUDAExtension` runs `torch.utils.hipify` on the `.cu`/`.cuh`/`.h` at build time and links amdhip64/c10_hip/torch_hip. Keep all sources in CUDA spelling; do NOT add a compat header or hand-rename symbols. Fix ONLY what hipify cannot (warp-size, sync-mask width, HIP CG / cub gaps, nvcc-only flags), guarded by `USE_ROCM`. This is the gaussian_splatting / gsplat / LiteGS model. Incremental-build gotcha (Strategy B): after editing a `.cu`, delete the extension's `build/` and any `*.hip`/`*_hip.cuh` hipify mirror before rebuilding, or torch recompiles the stale mirror; gitignore those artifacts.

Environment (verified present per task): torch 2.13 ROCm (torch.version.hip ~7.2.5), conda env py_3.12, gfx90a MI250X, ROCm 7.2.1. MUST build/run with cwd OUTSIDE /var/lib/jenkins/pytorch (its tree shadows the installed torch and breaks CUDAExtension hipify). Cap concurrent GPU agents at 4 (4 GCDs); pin one GCD via HIP_VISIBLE_DEVICES.

## CUDA surface inventory

### (a) RASTERIZER classification: wave32-HARDCODED (LiteGS-class, HARD) -- NOT the easy gsplat/gaussian_splatting family
This is a fork of the inria base, but it has been re-engineered with explicit per-warp logic (the upstream stock diff-gaussian-rasterization backward is shared-mem + __syncthreads with no warp intrinsics; this fork is not). The hard parts:
- `auxiliary.h:19` `#define GSGN_NUM_WARPS (GSGN_BLOCK_SIZE/32)` -- hardcoded /32 warp count (block is 16x16=256).
- `backward.cu` `renderCUDABW_butterfly` / `renderCUDABW_serialized` (3 backward variants, launched by a runtime selector at 1148/1170/1193): `laneId = (threadIdx.y*blockDim.x+threadIdx.x) & 31`, `__match_any_sync(__activemask(), idx)`, `__popc`, `__ffs`, `(1<<leader)|(1<<target_lane)` 32-bit lane-mask arithmetic, `__shfl_down_sync(0xFFFFFFFF,...)` butterfly over 9 grads, `__ballot_sync(__activemask(), !skip)` + `__popc`, `__match_any_sync(...)==0xFFFFFFFF`, `__syncwarp()`.
- `gsgn.cu` (THE new LM file, 2765 lines): pervasive `lane_id = threadIdx.x % 32`, `warp_id = threadIdx.x / 32`, `NUM_WARPS = SPARSE_J_NUM_THREADS/32` (and `.../32` elsewhere), `#define FULL_MASK 0xffffffff`, `__ballot_sync(FULL_MASK, idx<stride)`, `__shfl_up_sync`, `cub::WarpScan<int>`, `cub::WarpReduce<scalar_t>::HeadSegmentedSum`, `__match_any_sync`/`__shfl_sync` (atomred_vec), shared arrays sized `[NUM_WARPS][32]`, and "256-byte read = 8 byte/thread * 32 threads" coalescing assumptions (gsgn.cu:1784).

### (b) THE LM OPTIMIZER (the distinguishing part) -- solver backend mapped precisely
HOW the normal equations (J^T J + lambda I) dx = J^T r are solved: **matrix-free batched Preconditioned Conjugate Gradient in pure PyTorch**, NOT cuSPARSE / cuSOLVER / cuBLAS.
- `cg_batch.py`: a self-contained batched PCG (`cg_batch(A_bmm, B, M_bmm, ...)`) built ENTIRELY from torch tensor ops (`.sum(1)`, `torch.norm`, broadcasted add/mul). No torch.linalg, no cholesky/lstsq, no cusolver/cusparse/cublas symbols anywhere in the repo (grep-confirmed).
- `train.py linear_solve_pcg_fused` wires it up: `A_bmm(X) = apply_jtj(...)` (the custom JtJ matvec kernel) and `M_bmm(X)` = the Jacobi preconditioner from `calc_preconditioner` (the `diagJTJ` kernel), with a trust-region `(1/radius)` damping added on the diagonal. `lm_step` runs PCG, line-searches gamma, accepts/reverts via trust-region update.
- The GPU-custom pieces (all in gsgn.cu, exposed via the python bindings) are: `eval_jtf_and_get_sparse_jacobian` (paper buildCache: builds the sparse-Jacobian cache + b=J^T F), `sort_sparse_jacobians`/`gsgn_sort_sparse_jacobians` (sortCacheByGaussians, uses `cub::WarpReduce` HeadSegmentedSum), `calc_preconditioner`/`apply_jt` in PRECONDITIONER mode (diagJTJ), `apply_jtj` = `apply_j` (applyJ) then `apply_jt` (applyJT) with an `apply_j_resort_x_vec_kernel` (sortX) in between.

PORT IMPLICATIONS of the solver path (the key good news): because the solve is torch-native PCG, there is NO sparse/dense linear-algebra backend to swap -- the arrayfire/amgcl hipSPARSE precedent and the raft cuSOLVER-*Host-gap precedent DO NOT APPLY here. PCG itself runs free on ROCm-torch. ALL the LM risk collapses onto the custom JVP kernels in gsgn.cu, which are exactly the wave32-hardcoded reductions in (a). So the LM optimizer reduces to "the same wave64 reduction fault class as the rasterizer," not a library-coverage problem.

### Libraries / other surface
- cub/hipCUB: `rasterizer_impl.cu` `cub::DeviceScan::InclusiveSum` + `cub::DeviceRadixSort::SortPairs` (begin_bit=0 -> the cudaKDTree nonzero-begin_bit bug does NOT apply); `simple_knn.cu` `cub::DeviceReduce::Reduce` (CustomMin/Max) + `DeviceRadixSort::SortPairs` (begin_bit=0) + thrust device_vector/sequence/sort. hipify maps `<cub/cub.cuh>`/`<cub/device/...>` to hipCUB headers but leaves the `cub::` NAMESPACE unrenamed -> add `namespace cub = hipcub;` under USE_ROCM after the include (gsplat fault 6 / cudaKDTree). rocThrust is a drop-in (no source swap).
- cub WARP collectives: `cub::WarpScan<int>` and `cub::WarpReduce<scalar_t>` default `LOGICAL_WARP_THREADS = warpSize` -- 32 on CUDA, 64 on hipCUB/gfx90a. This is the single most dangerous swap (see Risk list).
- Cooperative groups: `<cooperative_groups.h>` (mapped) + `<cooperative_groups/reduce.h>` (forward.cu:15, rasterizer_impl.cu:28). HIP 7.2.1 has NO `<cooperative_groups/reduce.h>` and no `cg::reduce`. BUT here CG is used only for `cg::this_grid().thread_rank()` and `cg::this_thread_block()` (both provided by HIP CG) -- NOT for cg::reduce. So `#include <cooperative_groups/reduce.h>` must be USE_ROCM-guarded out (hipify leaves it verbatim), but no cg::reduce butterfly replacement is needed (unlike gaussian_splatting). Verify with a grep for `cg::reduce` (expected: none).
- Managed memory: `cudaMallocManaged` for arrays-of-device-pointers in rasterizer_impl.cu, gsgn_data_spec.h, rasterize_points.cu (host fills pointer values, device reads them). hipMallocManaged works on gfx90a. The cudaKDTree "int atomicMin/Max silently dropped on coarse-grained managed memory" fault does NOT apply: these managed buffers are pointer tables, never atomic-RMW targets. Keep as managed (portable); note only.
- half: `__half`/`__float2half`/`__half2float` for the packed sparse-Jacobian cache (GradientCache). HIP provides all of these; torch's COMMON_HIPCC_FLAGS sets `-D__HIP_NO_HALF_OPERATORS__=1`/`-D__HIP_NO_HALF_CONVERSIONS__=1` -- if gsgn.cu uses half operators/implicit conversions (it uses explicit __float2half/__half2float, likely fine) and the build errors, strip those two flags in setup.py (the LiteGS fix). Apply only if needed.
- `--use_fast_math` (rasterizer setup.py only): nvcc-only; hipcc/clang rejects it. Guard to `-ffast-math` on ROCm OR drop it (watch the gsplat fault-13 fast-math gradient-tolerance perturbation -- prefer dropping fast-math on ROCm and relying on the PSNR/loss gate; do NOT chase bit-exactness). `--generate-line-info`/`-lineinfo` are also nvcc-only -> guard/drop on ROCm.
- simple-knn `#define __CUDACC__` (simple_knn.cu:23) -- an editor-intellisense force-define; on hipcc this can steer headers (GLM, the MPPI fault). simple-knn includes GLM? (it includes its own headers; check). If __CUDACC__ being defined disturbs the HIP build, USE_ROCM-guard it out (LiteGS did exactly this).
- Textures/surfaces: NONE (grep-confirmed: no tex*/surf*/cudaArray). 256B-pitch, layered-array, and linear-filter fault classes are all N/A.
- `__trap()` (auxiliary.h:320, in_frustum prefiltered assert): HIP supports __trap(); fine.
- simple-knn `boxMinMax` shared-memory min/max tree (simple_knn.cu:96): folds from BOX_SIZE/2=512 down to 1 with a `__syncthreads()` BEFORE AND AFTER every step (no unsynced warp-synchronous tail) -> the MPPI wave64 unsynced-reduction fault does NOT apply; correct on wave64 as-is.

## Risk list (ranked; gfx90a wave64 = 64 lanes is the dominant axis)

1. **cub::WarpReduce / cub::WarpScan default to a 64-wide logical warp on hipCUB (HIGH, correctness).** `cub::WarpReduce<scalar_t>` (gsgn.cu:1738, 2293) and `cub::WarpScan<int>` (gsgn.cu:177) use `LOGICAL_WARP_THREADS = warpSize`. On gfx90a that is 64, but the surrounding code partitions threads into 32-lane warps (`lane_id=tid%32`, `warp_id=tid/32`, `temp_storage[NUM_WARPS]` with `NUM_WARPS=threads/32`, `[NUM_WARPS][32]` shared). A 64-wide collective with per-32 temp-storage indexing => two logical 32-warps share one TempStorage slot and the collective spans both => wrong sums AND a TempStorage race (the CV-CUDA wave64 TempStorage class). FIX: pin the logical width to 32 explicitly: `cub::WarpReduce<scalar_t, 32>` / `cub::WarpScan<int, 32>` (both take a `LOGICAL_WARP_THREADS` template arg). Keep 32 on BOTH backends so the segment math, head-flag shuffles, and `[NUM_WARPS][32]` layout stay consistent. This is THE load-bearing LM-kernel fix.

2. **`__shfl_up_sync(mask, global_id, 1)` head-flag segmented reduction must stay 32-wide (HIGH, correctness).** gsgn.cu:1776/2317 detect Gaussian-segment heads within a warp via shfl_up. The mask is `__ballot_sync(FULL_MASK=0xffffffff, idx<stride)`. On HIP the *_sync builtins REQUIRE a 64-bit mask (ROCm 7.x static_asserts sizeof(mask)==8) -- the 32-bit `0xffffffff` literal fails to COMPILE (AutoDock-GPU lesson). FIX: a USE_ROCM full-warp mask `0xffffffffffffffffULL`. BUT the wave64 ballot then spans 64 lanes while WarpReduce<,32> spans 32 -- the head-flag (lane_id==0 || id!=prev) and the segmented sum must use the SAME 32-lane grouping. Simplest correct approach: make the whole kernel treat the wavefront as TWO independent 32-lane groups (the popsift/LiteGS positional-packing model): keep `lane_id = threadIdx.x % 32`, drive the segmented reduce with `cub::WarpReduce<,32>`, and confine shfl_up to width-32 (`__shfl_up_sync(mask, v, 1, 32)` or a 32-wide CG tile). Verify the ballot mask is consumed only per-32 (the head_flag for lane 32 must come from its own group's prev, not lane 31's value). This kernel family (eval_jtf buildCache + apply_jt diagJTJ/applyJT + sort) is the highest-effort part of the whole port.

3. **`__match_any_sync` / `__activemask` / `__ballot_sync` + 32-bit lane masks in backward.cu and gsgn.cu atomred_vec (HIGH, correctness + compile).** `atomred_vec` (gsgn.cu:92) and the backward butterfly (backward.cu:414-448, 763-840) do per-primitive leader election: `same_mask = __match_any_sync(__activemask(), idx)`, `__popc(same_mask)`, `__ffs`, `(1<<leader)|(1<<target_lane)`, `if(__match_any_sync(...)==0xFFFFFFFF ...)`. On wave64: (i) the 32-bit `0xffffffff`/`(1<<lane)` mask arithmetic is wrong for 64 lanes and the `==0xFFFFFFFF` "full-warp" test never fires; (ii) `__popc` on a 64-bit ballot drops the upper half (the LiteGS compact.cu leader-election trap); (iii) `__ffs`/shift on a 64-bit mask needs `__ffsll`/64-bit shifts. FIX options: either confine to 32-lane groups (treat wavefront as 2x32, `__popc` per 32) OR go fully 64-bit (`__popcll`, `1ULL<<lane`, `__match_any_sync` returns 64-bit, `lane_id=tid&63`). Choose per-kernel to match how the surrounding `lane_id` is defined; keep it arch-unified via a `kWarpSize` constant (64 gfx90a / 32 RDNA+CUDA) where the kernel genuinely wants a full wavefront, and width-32 where it wants the NVIDIA-warp granularity. atomred_vec drives gradient scatter correctness; getting the leader count wrong double-/under-counts gradients (the gsplat fault-7 "match the CUDA atomic granularity" lesson).

4. **HIP *_sync builtins require a 64-bit mask -> the 32-bit `FULL_MASK 0xffffffff` literal is a COMPILE error before any correctness question (HIGH, compile).** Every `__ballot_sync`/`__shfl_*_sync`/`__match_any_sync`/`__any_sync` site (backward.cu + gsgn.cu, ~79 intrinsic hits combined). Define a USE_ROCM `FULL_MASK 0xffffffffffffffffULL` (and keep `__activemask()` which HIP provides and returns the 64-bit active set). This surfaces during hipcc's HOST compile pass of the .cu.

5. **Host-side per-warp data layout is hardcoded to 32 pixels (`pad(x,16,2)` + `unfold(...,2,16)...reshape(...,32)`) (HIGH, correctness coupling).** `diff_gaussian_rasterization/__init__.py:443-465` build `n_contrib_vol_rend_prefix_sum` by summing contributions over a 2x16 (HxW) = 32-pixel patch == "one warp", then prefix-summing. The kernel `eval_jtf_sparse_render_bkwd` consumes this with `NUM_WARPS = GSGN_BLOCK_X*GSGN_BLOCK_Y/32` and `n_contrib_vol_rend_prefix_sum[img_id][global_warp_id]` where `global_warp_id` is derived from `warp_id` (rank/32) and a 32-round inner loop (`rounds = (range.y-range.x+31)/32`, `progress = i*32 + lane_id`, `collected_id[NUM_WARPS][32]`). The Python 32 and the device 32 MUST agree. Cleanest: keep BOTH at 32 (treat each 64-lane wavefront as two 32-lane warps in this kernel; the 2x16 patch stays a 32-pixel "warp"). If instead converted to 64, the Python `unfold` patch geometry must change to 64 pixels too -- avoid; pin to 32. This is a 3DGS-LM-specific risk that does not appear in the simpler splatting ports.

6. **rocThrust functor returning a reference / lifetime UB (LOW-MED).** simple-knn CustomMin/CustomMax functors feed `cub::DeviceReduce::Reduce`; verify their `operator()` returns BY VALUE (the cudf cast_fn dangling-reference class). Likely fine (inria code returns MinMax by value), but check.

7. **fast-math gradient perturbation (LOW, gated away).** Dropping `--use_fast_math` on ROCm (or using -ffast-math) can shift ill-conditioned gradients ~ULP (gsplat fault 13). The validation gate is loss-down / PSNR-up + finite-difference slope, NOT bit-exactness, so absorb it; do not pin -ffp-contract chasing exactness on a training-convergence project.

8. **PYBIND11_MODULE on the gfx1151 follower (DEFERRED to delta-plan).** ext.cpp uses PYBIND11_MODULE + py::class_, which can hit the c10 inherited-ctor dllexport blocker on Windows gfx1151 (the fused-ssim wall). Irrelevant to gfx90a; noted for the follower.

## Test plan
Upstream ships NO test suite and NO requirements.txt/CI (grep-confirmed: no test_*.py/conftest). Define the gate (LiteGS Tier model + gsplat-style autograd checks):

Concrete definition of "validated on gfx90a":
1. BUILD: both extensions compile + link under ROCm for gfx90a (roc-obj-ls shows gfx90a code objects); `import simple_knn._C` and `import diff_gaussian_rasterization._C` succeed and all 9 ops register (rasterize_gaussians, rasterize_gaussians_backward, mark_visible, eval_jtf_and_get_sparse_jacobian, calc_preconditioner, apply_jtj, apply_j, sort_sparse_jacobians, filter_reordered_geometry_buffer).
2. RASTERIZER correctness (Tier 1): forward render of a small synthetic multi-Gaussian scene is finite, non-trivial, and bitwise-deterministic across 2 same-seed runs; backward via `torch.autograd.gradcheck`/central-finite-difference of `_RasterizeGaussians.apply` w.r.t. colors/opacity/means -- slope ~1.0, high sign agreement, residual within fp tolerance (the rasterizer backward scatters per-point grads via atomicAdd, so accept ~1e-7 atomic-reorder noise; gsplat/LiteGS bar).
3. LM-STEP correctness (Tier 2, the distinguishing gate): on a small scene, exercise ONE full `lm_step`:
   - `eval_jtf_and_get_sparse_jacobian` -> finite b=J^T r and a populated sparse-Jacobian cache;
   - PCG (`cg_batch` with A_bmm=apply_jtj, M_bmm=calc_preconditioner) CONVERGES: residual_norm decreases monotonically and `info["optimal"]` true (or residual << initial) within pcg_max_iter; the solve is deterministic across 2 runs;
   - VERIFY the matvec is a correct J^T J p: compare `apply_jtj(p)` against a reference `J^T (J p)` built from the rasterizer's own autograd (autograd jvp/vjp of the forward render) on a tiny scene -- relative error within fp/half tolerance (the cache stores half Jacobians, so expect ~1e-2..1e-3, target the largest entries). This is the decisive correctness oracle for the wave64 reduction fixes (risks 1-5): a wrong segmented sum makes JtJ wrong and PCG diverges or converges to the wrong dx.
4. END-TO-END (Tier 3, the headline gate): a short LM-optimized training run -- loss DOWN and PSNR UP over the LM iterations, no NaN, on a small scene. Prefer a real COLMAP scene if one is on the host (Mip-NeRF360 garden per the README command, with --iterations small e.g. 5-20 LM steps); else a self-contained synthetic multi-view fit through the full pipeline (LiteGS Tier-3 model: known ground-truth cameras + Gaussians, assert loss decreases and PSNR climbs). "Converging LM step" = at least one accepted step that reduces the cost and a trust-region radius that adapts; run twice for determinism of the accepted trajectory (allow ULP/atomic-order divergence, assert stable loss-down).
Non-GPU regression set: there are no upstream non-GPU unit tests to regress; the scene/colmap loaders and metrics are pure Python and exercised by Tier 3. Note any host-only script (render.py/metrics.py needs lpips weights, torchvision) that cannot run offline as a known non-blocker.

## Staged strategy + likely walls (keyed to precedents)
- Stage 0 -- build bring-up (compile walls, mechanical): the *_sync 64-bit-mask compile error (AutoDock), `cub::` namespace (gsplat/cudaKDTree), `<cooperative_groups/reduce.h>` guard (gaussian_splatting), nvcc-only flags + half-operator flags (LiteGS, gsplat), `#define __CUDACC__` (LiteGS/MPPI). Gets both .so to link and import.
- Stage 1 -- rasterizer wave64 correctness (backward.cu): the LiteGS-class leader-election/butterfly fixes; gate with Tier-2 gradcheck. Wall: getting `__match_any_sync`/`__popc`/lane-mask granularity right (LiteGS compact.cu + gsplat fault-7 atomic-granularity).
- Stage 2 -- THE LM kernels (gsgn.cu), the highest-risk wall: `cub::WarpReduce<,32>`/`WarpScan<,32>` + width-32 head-flag segmented reduce, kept consistent with the Python 32-pixel-warp layout. This is where a wrong wave64 reduction silently corrupts J^T J and PCG diverges -- the decisive oracle is Tier-2 (apply_jtj vs autograd J^T(Jp)) and Tier-3 convergence. Expect this to dominate the effort; budget the 3-attempt porter cap here.
- Stage 3 -- end-to-end Tier-3 + determinism; document the gate in notes.md.
The solver (PCG) is NOT a wall: torch-native, no hipSPARSE/hipSOLVER coverage question, runs free. The walls are entirely the custom JVP/segmented-reduce kernels' wave64 correctness -- treat 3DGS-LM as "LiteGS-hard rasterizer PLUS a second, larger LiteGS-hard kernel file (gsgn.cu)."
