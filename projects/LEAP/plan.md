# LEAP -- ROCm/HIP port plan (lead platform: linux-gfx90a)

## Project
- Name: LEAP (LivermorE AI Projector for Computed Tomography)
- Upstream: https://github.com/llnl/LEAP
- Default branch: main
- Head at clone: 0c8846f ("updated version")
- What it is: a C++/CUDA library of 3D tomographic CT algorithms (forward/back
  projectors for parallel/fan/cone/cone-parallel/modular beam, FBP/FDK, iterative
  recon, ramp filtering, denoisers) with a Python (ctypes) interface `leapctype`
  and a PyTorch autograd binding `leaptorch`. The GPU work is a large set of
  hand-written projection-geometry kernels that sample the volume/sinogram
  through hardware-interpolated 3D textures. Multi-GPU + multi-core CPU paths
  both exist (the CPU `_cpu.cpp` files mirror every GPU kernel and are the natural
  gold reference).

## Existing AMD support -> abandoned / incomplete HIP port; FINISH it
- README "Future Releases" lists "AMD GPU Support" as not-yet-done.
- The tree carries hipify leftovers that are NOT wired into any build and not
  included by any source: `src/hip_utils.h` (header banner literally says
  "automatically generated by hipify"), `src/projectors_Joseph_cpu_hip.h`,
  `src/ramp_filter_hip.cuh`. No `.cu`/`.cpp` includes them; no build script
  references them. They are a stale one-shot hipify dump, not a working path.
- `setup_AMD.py` exists but is byte-for-byte equivalent in behaviour to
  `setup.py` (the `rocm` and `cuda` branches set identical flags); it adds no
  real ROCm handling. It is a placeholder.
- There is NO OpenCL/Vulkan/SYCL AMD path; the only GPU backend is CUDA.
- Decision: PROCEED. This is an abandoned/incomplete HIP attempt. Value = deliver
  a real, GPU-validated ROCm port. We will NOT resurrect the orphan `*_hip.*`
  files (they would desync from the live `.cu` sources); we rely on torch's
  build-time hipify (Strategy B) over the canonical CUDA sources and fix the one
  real fault class in-source under `__HIPCC__`/`USE_ROCM` guards.

## Build classification -> torch extension (Strategy B)
Evidence:
- `setup.py` / `setup_AMD.py` import `torch.utils.cpp_extension`
  (`CUDAExtension`, `BuildExtension`) and build a single `leapct` extension from
  the 21 `cpp_files` + 19 `cuda_files`. This is the installed Python module
  (`pip install .` runs this; see Dockerfile `RUN pip install .`).
- A second, pytorch-FREE path exists (`setup_ctype.py` -> `etc/build.sh` ->
  top-level `CMakeLists.txt` -> `src/CMakeLists.txt`, `project(leapct CXX CUDA)`,
  `find_package(CUDA)`, builds `libleapct.so`). This is a pure-CMake CUDA build
  used by the ctypes-only install. It is a viable Strategy-A alternative, but the
  primary, documented install is the torch extension and the test suite imports
  `leapctype`/`leaptorch` against the built `leapct` module.
- Chosen path: build the torch `CUDAExtension` on a ROCm PyTorch. On ROCm torch,
  `CUDAExtension` auto-runs `torch.utils.hipify` on the `.cu`/`.cuh` sources and
  links amdhip64/c10_hip/torch_hip. Keep sources in CUDA spelling; do NOT add a
  compat header or hand-rename symbols (Strategy B rule).

## Port strategy -- B (torch build-time hipify) + targeted in-source fixes
Rationale: the project is fundamentally a torch extension and ships its own
`.cu`; the minimal-footprint approach is to let ROCm torch hipify at build time
and fix only what hipify cannot. The kernels are plain projection-geometry math
with NO CUTLASS/CuTe, NO Hopper PTX, NO wgmma, NO warp specialization, NO cub/
thrust, NO cooperative groups, NO warp intrinsics at all (grep for
`__shfl*`/`__ballot*`/`warpSize`/`tiled_partition` returns nothing). So this is a
portable hand-written CUDA port, NOT a reimplement-not-port case. A mechanical
HIP port is correct and sufficient; no AMD-native rewrite is warranted for a
first delivery.

Delete-from-build / ignore the orphan `*_hip.*` files; do not include them.

## CUDA surface inventory
- Kernels: ~146 `__global__` kernels across 20 `.cu` files (projectors_SF 15,
  ramp_filter 15, projectors_extendedSF 15, backprojectors_VD 12, projectors_Joseph
  11, projectors_Siddon 8, projectors_symmetric 6, sensitivity 5,
  projectors_attenuated 5, scatter_models 4, bilateral_filter 4, noise_filters 13,
  ray_weighting 3, total_variation 3, cuda_utils 28 elementwise helpers, etc).
  All are ordinary index-mapped kernels; no block-collective / warp-collective use.
- Textures (DOMINANT surface): central helpers in `cuda_utils.cu` --
  `loadTexture` (3D, `cudaMalloc3DArray` + `cudaResourceTypeArray`),
  `loadTexture1D`/`loadTexture2D` (`cudaMallocArray`), `loadTexture_from_cpu`.
  All bind float `cudaArray` textures with `readMode = cudaReadModeElementType`,
  `normalizedCoords = false`, addressMode Clamp or Border, and filterMode chosen
  by a `useLinearInterpolation` flag -> `cudaFilterModeLinear` when true.
  ~700+ `tex3D<float>/tex2D<float>/tex1D<float>` fetches across the projectors,
  the vast majority at FRACTIONAL coordinates relying on HW trilinear/bilinear
  interpolation. Many hot paths pass `useLinearInterpolation = true`
  (backprojectors_VD, projectors_SF/extendedSF, ramp_filter row-resample,
  geometric_calibration); some pass false (point-sample: bilateral/guided/noise
  filters, matching_pursuit, some volume binds).
- No surfaces (`surf*`), no layered arrays, no managed memory, no pinned host
  alloc, no explicit streams/events on the hot path (one commented-out
  `cudaStreamCreate`), no fp16/__half, no atomics, no cooperative groups.
- Libraries: cuFFT only (`ramp_filter.cu`, guarded by `__INCLUDE_CUFFT`):
  `cufftPlan1d`, `cufftExecR2C`/`C2R`, `cufftComplex`/`cufftReal`,
  `cufftDestroy`, R2C/C2R 1D batched transforms for ramp/Hilbert filtering.
  No cuBLAS/cuRAND/cuSPARSE/Thrust/CUB despite the CMake linking cublas (the
  CMake links `${CUDA_cublas_LIBRARY}` but no source calls cuBLAS -- dead link).
- Host GPU mgmt: `cudaMalloc/Free/Memcpy/Memcpy3D(Async)`, `cudaSetDevice`,
  `cudaDeviceSynchronize`, `cudaGetDeviceProperties`, `cudaMemGetInfo`,
  device enumeration -- all 1:1 HIP via hipify.

### CUDA -> ROCm/HIP mapping
- `cudaMalloc/Free/Memcpy*/Memcpy3D*/cudaArray/cudaMalloc3DArray/cudaMallocArray`
  -> hip equivalents (hipify, 1:1).
- `cudaTextureObject_t`/`cudaCreateTextureObject`/`cudaTextureDesc`/
  `cudaResourceDesc`/`cudaChannelFormatDesc`/`tex3D`/`tex2D`/`tex1D`
  -> hipTextureObject_t etc (hipify, 1:1 API) -- BUT see Risk #1 (linear filter).
- cuFFT -> hipFFT: `cufftPlan1d`->`hipfftPlan1d`, `cufftExecR2C`->`hipfftExecR2C`,
  `cufftExecC2R`->`hipfftExecC2R`, `cufftComplex`->`hipfftComplex`,
  `cufftReal`->`hipfftReal`, `cufftDestroy`->`hipfftDestroy`, enums 1:1 for the
  codes used (R2C/C2R/SUCCESS). Link `hipfft`. torch hipify maps these names; the
  ROCm torch build must have hipFFT available (ROCm 7.2.1 ships it).
- `--use_fast_math` (CMake) -> on HIP that is `-ffast-math`; for the torch build
  pass nothing special (kept O3). Watch fp drift only if a test is bit-exact.

## Risk list (ranked)
1. **Hardware linear filtering on float element-read textures is REJECTED by HIP**
   (PORTING_GUIDE fault class; popsift/gpuRIR). Every `loadTexture*` that passes
   `useLinearInterpolation = true` will fail at `hipCreateTextureObject` with
   "operation not supported" -- AMD HW does not support linear filtering on
   `cudaReadModeElementType` float textures. This breaks the MAJORITY of the hot
   projector/backprojector paths and is THE central port task.
   Fix (matches the guide): on HIP create the texture `cudaFilterModePoint` and do
   the interpolation in software at the fetch site, matching CUDA's unnormalized
   -0.5 texel-center convention (coord c -> floor(c-0.5) and +1, weight
   frac(c-0.5)), trilinear for tex3D / bilinear for tex2D / linear for tex1D.
   Implementation choice to evaluate at port time: (a) route every `tex3D<float>`
   through a single project fetch helper (`texVal3D(tex,x,y,z)`) that does
   point-fetch-8 + trilerp on HIP and `tex3D` on CUDA, force-applied so the ~700
   call sites are untouched; OR (b) keep linear binds but only where HW supports
   them (none on float). Prefer (a). This is sizable but mechanical and is the
   bulk of the porter's work; correctness is checkable against the CPU path which
   already does its own SW interpolation.
2. **cuFFT -> hipFFT semantics**: batched 1D R2C/C2R sizing and the R2C output
   layout (N/2+1 complex) match between cuFFT and hipFFT, but verify the
   `cufftPlan1d` batch count path and that hipFFT does not require a separate work
   area. hipFFT status-enum coverage is not 1:1 (MPPI lesson) -- only the codes
   LEAP references (SUCCESS, plan create) are used, so low risk; do not blanket
   `#define` unused codes.
3. **Texture address-mode Border with float**: LEAP uses
   `cudaAddressModeBorder` for extrapolation=off (returns 0 outside). HIP point
   textures honor Border, but when we move to SW interpolation we must reproduce
   the border-zero / clamp behaviour in the fetch helper exactly (out-of-range
   neighbor -> 0 for Border, clamp index for Clamp), or edge voxels drift. This is
   also the OOB-neighbor-read fault class: SW interp reads c and c+1, so clamp/zero
   the +1 neighbor at the high edge (AMD faults on the past-end read CUDA tolerated).
4. **warpSize / wave64**: LOW. No warp intrinsics, no hardcoded 32, no
   block-collectives, no shared-memory warp reductions anywhere. The MULTI-ARCH
   warpSize standard still applies as a guard rail (build gfx90a;gfx1100 as the
   test) but no source warp-width work is expected. If the porter finds a hidden
   blockDim==32 assumption, fix per the per-arch constant rule; not anticipated.
5. **fast-math fp drift**: clang(HIP) defaults `-ffp-contract=fast`; if any test
   is bit-exact vs the CPU gold, pin `-ffp-contract=on`. CT recon tolerances are
   typically relative-error, so likely a non-issue; revisit only if a numeric
   test fails by ~1 ULP-scale.
6. **Orphan `*_hip.*` files drifting into the build**: ensure setup does not pick
   them up (they are headers; the `cuda_files` list does not include them, good).
   Leave them untouched; do not include them.
7. **`__USE_GPU` / `__USE_CPU` macro gating**: GPU build defines `__USE_GPU`; the
   `.cu` are only compiled when `torch.cuda.is_available()`. On the ROCm box torch
   sees the AMD GPU as "cuda", so the GPU branch is taken automatically. Confirm
   `torch.cuda.get_device_name(0)` contains "AMD" so the `rocm` setup branch fires
   (it is currently a no-op, but harmless).

## File-by-file change list (anticipated; porter confirms)
- `setup.py` (and/or `setup_AMD.py`): ensure the ROCm branch builds cleanly under
  ROCm torch; add `hipfft` to libraries on ROCm if hipify does not auto-link it
  for the ramp filter. Likely keep `setup.py` as the single entry and let hipify
  do the work. Minimal edits.
- `src/cuda_utils.cu` / `src/cuda_utils.h`: in `loadTexture`, `loadTexture1D`,
  `loadTexture2D`, `loadTexture_from_cpu`: under HIP, force `cudaFilterModePoint`
  regardless of `useLinearInterpolation`, and record per-texture whether SW
  interpolation should be applied (so the fetch helper knows). Provide the
  device-side `__USE_ROCM__`-guarded SW-interp fetch helpers
  (`texVal3D/texVal2D/texVal1D`) that on CUDA forward to `tex*D<float>` and on HIP
  point-fetch + lerp with the -0.5 convention and Border/Clamp handling.
- `src/projectors_*.cu`, `src/backprojectors_VD.cu`, `src/ramp_filter.cu`,
  `src/sensitivity.cu`, `src/scatter_models.cu`, `src/geometric_calibration.cu`,
  `src/resample.cu`, `src/matching_pursuit.cu`, `src/noise_filters.cu`,
  `src/bilateral_filter.cu`, `src/guided_filter.cu`, `src/total_variation.cu`,
  `src/analytic_ray_tracing_gpu.cu`, `src/projectors_attenuated.cu`,
  `src/ray_weighting.cu`: replace direct `tex3D<float>(...)`/`tex2D`/`tex1D`
  fetches at fractional coords with the project fetch helper so the SW-interp
  path is taken on HIP. (The point-sampled binds -- `useLinearInterpolation=false`
  -- need no math change, only the forced point filter mode, which they already
  effectively want.) This is the large mechanical edit; scope it to the
  linear-interp fetch sites first, verify, then sweep the rest.
- `src/ramp_filter.cu`: confirm `__INCLUDE_CUFFT` path compiles under hipFFT (it
  is the only cuFFT user). The torch build does not currently define
  `__INCLUDE_CUFFT`; decide whether the ramp filter needs the GPU FFT path on
  ROCm (CPU FFTW fallback exists for the ctype build). For first validation the
  projector forward/back slice does not require cuFFT; FBP/ramp validation needs
  hipFFT wired in.
- Do NOT add a compat header (Strategy B). Guards use `USE_ROCM` (torch defines
  it) or `__HIP_PLATFORM_AMD__`.

## Build commands (gfx90a, lead)
Build against a ROCm PyTorch (ROCm 7.2.1). From a fresh conda/venv with ROCm torch:
```
cd projects/LEAP/src
# confirm torch sees the AMD GPU as cuda:
python -c "import torch; print(torch.cuda.is_available(), torch.cuda.get_device_name(0))"
# build + install the extension (hipify runs automatically on the .cu):
PYTORCH_ROCM_ARCH=gfx90a python setup.py build_ext --inplace 2>&1 | tee build.log
# or: PYTORCH_ROCM_ARCH=gfx90a pip install -e . --no-build-isolation
```
For the multi-arch warp-size sanity build (the standard's correctness test), set
`PYTORCH_ROCM_ARCH="gfx90a;gfx1100"` and confirm both code objects:
`llvm-objdump --offloading build/.../leapct*.so | grep -E "gfx90a|gfx1100"`.
(LEAP has no warp-width source code, so this is a regression guard, not a fix gate.)

If the CMake/ctype path is preferred for an isolated GPU smoke, a Strategy-A
overlay (`enable_language(HIP)`, mark `.cu` LANGUAGE HIP, default arch from
`${CMAKE_HIP_ARCHITECTURES}`) is a documented fallback but is NOT the chosen
primary path.

## Test plan
- GPU-validatable slice (primary correctness gate): `unitTests/unit_tests.py`
  exercises all 6 geometries (parallel/fan/cone-parallel/cone flat/cone curved/
  modular) x voxel scales x VD and SF projection methods, running forward project
  + back project on GPU. The CPU projector (`*_cpu.cpp`, already built in the same
  module) is the in-tree gold reference; LEAP's own tests compare GPU vs CPU and
  vs analytic ray-tracing -- exactly the trilinear-interp correctness check we
  need. Add a fixed phantom forward/back round-trip and assert GPU matches CPU
  within the project's tolerance for each geometry.
- Secondary: `demo_leapctype/d01_standard_geometries.py` and
  `d02_standard_geometries_torch.py` (FDK/FBP recon of a phantom) -- exercises
  ramp filter + hipFFT + backprojection end to end; a recon-quality (SNR/RMSE)
  check is the FBP gate.
- `unitTests/memory_leak_tests.py`: regression for texture/array
  create/destroy churn (relevant since we touch every loadTexture).
- `unitTests/verificationTests.py` depends on an external `LTTserver` and a
  Windows-pathed object file (`C:\Users\champley\...`); it is NOT runnable here.
  Do not gate on it.
- Non-GPU regression set: the CPU-only path (`-D__USE_CPU`, no `.cu`) must still
  build and the CPU projectors must still pass the same unit_tests with
  `leapct.set_gpu(-1)` (CPU mode). Confirm we did not perturb CPU behaviour
  (we only edit `.cu` and HIP-guarded code).

## Open questions

  torch extension build (for the ramp-filter / FBP slice)? If not, scope first
  validation to the projector forward/back round-trip (no FFT) and add FBP after.

  hardware texture interp uses 8/9-bit fractional weights, so the existing CUDA
  HW path is already LESS precise than full-float SW interp; the HIP SW path may
  actually be CLOSER to the CPU gold than the CUDA HW path. Confirm the test

  exact must already pass with margin).

  Single-GPU validation is sufficient for the lead; note multi-GPU as a follow-up.
