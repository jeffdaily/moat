# CatBoost ROCm/HIP Port Plan (lead platform: linux-gfx90a, MI250X, ROCm 7.2.1)

## Project
- Name: catboost
- Upstream: https://github.com/catboost/catboost
- Default branch: master
- What it is: Yandex gradient-boosting-over-decision-trees ML library (Python/R/Java/C++/Spark). Ships a CUDA GPU training+inference engine ("Fast GPU and multi-GPU support"). ~96K LOC of GPU code under `catboost/cuda/` + `library/cpp/cuda/`.

## Existing AMD support: NONE -> PROCEED
Grep of the entire `catboost/cuda`, `library/cpp/cuda`, all `*.cmake`/CMakeLists, and `build_native.py` for `hip|rocm|hipify|__HIP__|gfx9|amdgpu` returns zero hits. No upstream ROCm/HIP/AMD issue or PR (searched). GPU support is CUDA-only (NVIDIA driver >= 450 required per docs). There is no OpenCL/Vulkan/SYCL path either. A ROCm/HIP port of the CUDA engine is genuinely new and high-value: CatBoost is a top-tier, widely-deployed ML library with no AMD GPU path today. **Decision: proceed with a correctness-first mechanical Strategy-A port.**

Port-vs-rewrite note: the GBDT histogram kernels (`pointwise_hist2_*`, `pairwise_hist_*`, greedy_subsets_searcher `hist_*`) are perf-critical and NVIDIA-32-lane-tuned (shared-memory sub-histograms, warp-aligned loads), but they are NOT CUTLASS/CuTe/wgmma -- they are hand-written shared-memory + shuffle kernels. A mechanical wave64-correct HIP port is the right first step (a later AMD-native MFMA/CK pass is not warranted; CatBoost has no GEMM/attention core). Correctness on gfx90a is the goal; perf tuning is out of scope for the initial port.

## Build classification: pure CMake -> Strategy A
Evidence:
- Root `CMakeLists.txt` is `project(CATBOOST LANGUAGES C CXX ASM)` and dispatches to per-platform `CMakeLists.<os>-<arch>-cuda.txt` gated on `HAVE_CUDA` (lines 58-86). No `find_package(Torch)`, no `torch.utils.cpp_extension`, no `CUDAExtension` anywhere in the tree (confirmed by grep -> empty).
- `cmake/cuda.cmake` does the CUDA enablement: `enable_language(CUDA)`, `find_package(CUDAToolkit REQUIRED)`, and defines the project's own `target_cuda_sources()/target_cuda_flags()/target_cuda_cflags()` helpers that attach nvcc `--compiler-options` flags per source. `.cu` files are added via `target_cuda_sources(<tgt> PRIVATE ...)`.
- The Python package (`catboost/python-package/setup.py --with-cuda/--no-cuda`) is a thin wrapper that drives the SAME CMake `HAVE_CUDA` build (it is not a cpp_extension build).
- Build entrypoints: `build/build_native.py --have-cuda --cuda-root-dir=...`, or direct `cmake -DHAVE_CUDA=yes -DCUDAToolkit_ROOT=... [CUDAARCHS=...]`.

CRITICAL CONSTRAINT (ya-make): every `CMakeLists*.txt` carries a header stating the files are GENERATED from Yandex `ya.make` and "only simple modifications are allowed (adding source-files, target_include_directories, simple properties); complex modifications which can't be easily ported back to ya may be rejected." This shapes the upstream-PR strategy (see Open questions). For the MOAT fork we will modify the generated CMake as needed to build on ROCm; the upstream PR will need to land the equivalent in ya.make / be discussed with maintainers. The CMake edits we DO need are deliberately the "simple/additive" kind (a USE_HIP option, a sibling `cuda.cmake` HIP branch, `.cu` LANGUAGE retag).

## Port strategy: Strategy A, colmap model, adapted for the project's custom CUDA-CMake harness
Goal: only `.cu`/`.cuh` translation units see the HIP toolchain; host C++ (the other ~4600 CMakeLists' worth) is untouched; NVIDIA build stays byte-identical.

1. **Compat header** `library/cpp/cuda/wrappers/cuda_to_hip.h` (single source of HIP knowledge). On `USE_HIP`/`__HIP_PLATFORM_AMD__`: `#include <hip/hip_runtime.h>` and alias the ~60 cuda* runtime symbols the engine uses (full list inventoried below) to their hip* equivalents (`cudaMalloc`->hipMalloc, `cudaStreamBeginCapture`->hipStreamBeginCapture, `cudaGraph*`->hipGraph*, `cudaEvent*`->hipEvent*, `cudaMemcpyKind`/enums, etc.). All have 1:1 HIP spellings (verified against the symbol dump). Generate the alias block from pytorch hipify `cuda_to_hip_mappings.py` filtered to the grep'd symbol surface. On NVIDIA the header is a no-op `#include <cuda_runtime.h>`. Include `<cstring>/<cstdlib>` BEFORE the HIP runtime (gpuRIR lesson: host memcpy/memset can resolve to HIP __device__ overloads otherwise).

2. **Harness: add a HIP branch to `cmake/cuda.cmake`** under `if(USE_HIP)` rather than rewriting it. The existing file is deeply nvcc-specific (`--expt-extended-lambda`, `--expt-relaxed-constexpr`, `--compiler-options`, `--threads`, `find_package(CUDAToolkit)`, vendored Thrust/CUB include dirs). Mirror it: when `USE_HIP`, `enable_language(HIP)`, set `CMAKE_HIP_ARCHITECTURES` from the cache var (default `gfx90a` ONLY when unset -- never a literal, per CudaSift/Gpufit lesson, so gfx1100/gfx1151 followers need no source edit), and redefine `target_cuda_sources()` to mark its sources `LANGUAGE HIP` + set `HIP_ARCHITECTURES` (so the dozens of `target_cuda_sources(...)` call sites in the per-dir CMakeLists are UNCHANGED -- this is the MPPI add_executable-override trick applied to the project's own wrapper function, zero churn in subdir CMake). Map the project's nvcc `--compiler-options` translation to hipcc-appropriate flags (mostly drop; hipcc takes host flags directly). Wire root `CMakeLists.txt` to include the cuda module / new `CMakeLists.linux-x86_64-hip.txt` variants OR (cleaner) reuse the existing `*-cuda.txt` source lists under a `HAVE_CUDA OR HAVE_HIP` condition so the .cu file lists are not duplicated.

3. **rocThrust/hipCUB redirect.** The build serves Thrust/CUB from vendored `contrib/deprecated/nvidia/{thrust,cub}` (CCCL 2.1.0) via `CUDA_EXTRA_INCLUDE_DIRECTORIES`. On HIP do NOT put those on the include path; rocThrust + hipCUB are header-only on `/opt/rocm/include` (hipcc default search path). rocThrust is a true `thrust::` drop-in (cudaKDTree lesson); for `cub::` add `#define cub hipcub` + include the hipcub umbrella in the compat header, OR rely on hipify-style include of `<hipcub/hipcub.hpp>`. The engine uses CUB device primitives (`DeviceScan`, `DeviceRadixSort`, `DeviceReduce`, `DeviceSegmentedRadixSort`, `BlockRadixSort`, `WarpScan`) and `cub::ThreadLoad/ThreadStore<LOAD_LDG/LOAD_CS/STORE_WT/STORE_CS>` -> hipCUB mirrors these.

4. Guard genuinely divergent code with `#if defined(USE_HIP)`; keep rare.

## CUDA surface inventory
- **GPU runtime layer** (`catboost/cuda/cuda_lib/` + `library/cpp/cuda/wrappers/`): ~60 cuda* runtime symbols, all 1:1 HIP: cudaMalloc/Free/FreeHost, cudaHostAlloc(Portable), cudaMallocManaged, cudaMemcpy(Async)/MemsetAsync + all cudaMemcpyKind enums, cudaStream* (Create/WithFlags/WithPriority/Destroy/Synchronize/WaitEvent/BeginCapture/EndCapture, cudaStreamPerThread, cudaStreamNonBlocking, cudaStreamCaptureModeThreadLocal), cudaEvent* (Create/WithFlags/Record/Query/Synchronize/Destroy, BlockingSync/DisableTiming), cudaGraph*/cudaGraphExec* (Instantiate/Launch/Destroy), cudaGetDevice(Count)/SetDevice/GetDeviceProperties/cudaDeviceProp/cudaDeviceSynchronize/DeviceGetStreamPriorityRange, cudaPointerGetAttributes/cudaPointerAttributes/cudaMemoryType*, cudaMemGetInfo, cudaGetLastError/GetErrorString, cudaError/cudaSuccess/cudaErrorNotReady.
- **CUDA Graphs + stream capture** (`cuda_graph.{h,cpp}`, `stream_capture.{h,cpp}`, `cuda_event.{h,cpp}`): RAII ref-counted handles -> hipGraph/hipStreamCapture. HIP supports graphs; watch the rule-of-five (below).
- **Kernels**: 82 `.cu`, ~450 `.cuh`. Hot path = histogram builders (`catboost/cuda/methods/kernel/pointwise_hist2_*`, `pairwise_hist_*`, `split_pairwise.cu`; `catboost/cuda/methods/greedy_subsets_searcher/kernel/hist_*`, `compute_hist_loop_*`), plus `cuda_util/kernel/` primitives (reduce, sort, scan, segmented_*, partitions, transform, fill, reorder).
- **Warp intrinsics / shuffles**: `__shfl_down_sync`, `__syncwarp` in `cuda_util/kernel/kernel_helpers.cuh`, `library/cpp/cuda/wrappers/kernel_helpers.cuh`, `targets/kernel/{pfound_f,yeti_rank_pointwise,dcg}.cu`, `methods/greedy_subsets_searcher/kernel/histogram_utils.cu`, `methods/kernel/linear_solver.cu`, `cuda_util/kernel/random_gen.cuh`. ~50 occurrences. Mask literals `0xFFFFFFFF` AND a latent-buggy `0xFFFFFF` (24-bit!) in WarpReduceN.
- **Cooperative groups**: `tiled_partition<8|16|32>` + `thread_block_tile<8|16|32>` with `tile.shfl_down`/`tile.sync` (TileReduce/TileReduce4/TTileReducer). All tiles <=32. Maps to `<hip/hip_cooperative_groups.h>`. NO use of cg::reduce/labeled_partition (hand-rolled butterfly), so the gsplat/cuCollections CG gaps mostly do not bite -- but `TTileReducer<64/128/256>` bodies are `asm("trap;")`.
- **Atomics**: atomicAdd x53 (incl. double CAS-fallback under `__CUDA_ARCH__<600`, and atomicAdd_block under >=600), atomicCAS x6, atomicOr x1, AtomicFMA via atomicCAS. NO atomicMin/atomicMax (avoids the cudaKDTree coarse-grained-memory drop fault). All used ops (Add/CAS/Or) are unaffected by that fault on managed memory.
- **Inline PTX** (only 2 files, 6 sites): `cuda_util/kernel/instructions.cuh` (`bfe.u32`, `ld.global.cg.{u32,v2.u32,f32}`, `prefetch.global.L1`) and `wrappers/kernel_helpers.cuh` (`asm("trap;")`). All need HIP replacements.
- **Hand-rolled vector operators**: full set of `float2`/`float4` operators (+ - * / and compound-assign, vector-vector AND vector-scalar) in `wrappers/kernel_helpers.cuh`. These COLLIDE with HIP's HIP_vector_type built-in operators.
- **Libraries**: Thrust (2 files) + CUB (21 files) -> rocThrust/hipCUB (header drop-ins, NO MOAT dep). cuSOLVER: present in source (`linear_cusolver.cuh`) BUT the OSS CMake build compiles `linear_cusolver_stub.cu` (empty `RunCholeskyCuSolver`), so cuSOLVER is NOT linked in the public build -- the Cholesky path falls back to the custom `linear_solver.cu` kernel. NO cuBLAS, cuRAND, cuFFT, cuSPARSE, cuDNN, NCCL anywhere. No CUDA driver-API (cuLaunch/cuModule/cuCtx) usage.
- **fp16**: only `targets/kernel/dcg.cu`. Minor.
- **Textures/surfaces**: NONE (the texture grep hits were `__ldg` read-only loads, trivially portable; no cudaTextureObject/surf*). This removes the entire colmap/popsift texture+layered-array fault family.
- **Managed memory**: `cuda_base.cpp`/`cuda_vec.h` use cudaMallocManaged. Low risk (only Add/CAS/Or atomics, which are coarse-grained-safe).

## Risk list
1. **wave64 warp-synchronous reductions (HIGH, the central risk).** Both `kernel_helpers.cuh` copies implement `WarpReduce`/`FastInBlockReduce`/`BlockReduceN`/`SharedReduce4`/`SharedReduce8`/`WarpReduceN` with the classic CUDA pattern: a `__syncthreads`-tree down to 32, then a 32-lane warp-synchronous tail (`s>=32` ... `if(threadIdx.x<32)` ... `__shfl_down_sync` / `__syncwarp` with `s=16,8,4,2,1`). On wave64 the low 32 lanes of a 64-lane wavefront are NOT lockstep across the unsynced shfl steps -> WRONG and non-deterministic (MPPI-Generic lesson). Cleanest fix per the guide: on HIP drop the warp-synchronous tail and let the `__syncthreads`-tree run to size 1 (same add order, correct on any wave size); for the shuffle-based `WarpReduce`, use width-aware shuffles and a 64-bit mask, OR convert to the shared-memory tree. This touches the GBDT hot path -> validate with fixed-seed bit-identical determinism on `test_reduce`/`test_scan`/`test_segmented_*`.
2. **Shuffle mask literals fail to COMPILE on HIP (HIGH, blocks build).** `__shfl_down_sync(0xFFFFFFFF,...)` and the buggy `0xFFFFFF` -- HIP's `*_sync` static_assert `sizeof(mask)==8` (AutoDock-GPU/raft lesson). Define a compat `FULL_WARP_MASK` (0xffffffffffffffffULL on HIP) keyed on USE_HIP, NOT on width. Surfaces in hipcc's host pass first.
3. **Hardcoded warpSize 32 / `& 31` / 32-lane histogram layout (HIGH).** No central warpSize abstraction exists (`TArchProps` exposes SM count + major/minor + fp16, but NOT warpSize). `compute_pair_hist_loop.cuh` has `const ui32 warpSize = 32` driving warp-aligned loads; `update_part_props.cu` has `const int warpSize=32` + `& 31`; histogram kernels build per-warp sub-histograms sized to 32 lanes. Introduce a `kWarpSize` per-arch constant (64 on `__GFX9__`, else 32; guide's C10_WARP_SIZE pattern) AND/OR a runtime `prop.warpSize` query in host launch math. Per-kernel decision (popsift vs AutoDock lesson): for histogram kernels that pack/align to 32-lane warps positionally, treat the wave as two 32-lane half-warps; for atomicAdd-combined reductions, a native 64-lane reduction is fewer steps. This is where most of the porter time goes.
4. **Hand-rolled float2/float4 operators collide with HIP vector types (MED, blocks build).** HIP's HIP_vector_type<float,N> ships the full operator set; the project's `operator+/*/...` on float2/float4 become "ambiguous" (Fast-Poisson/cupoch/MPPI lesson). `#if !defined(USE_HIP)`-guard each colliding overload individually (HIP provides vector-vector AND vector-scalar for +-*/, so all collide; keep non-operator helpers Dot4/FMA4/Sqrt4/Max4 which HIP lacks).
5. **Inline PTX (MED, blocks build, bounded).** `bfe.u32` -> bit ops `((a>>start)&((1u<<length)-1))` or `__builtin_amdgcn_ubfe`; `ld.global.cg.*` -> plain load or `__builtin_nontemporal_load` (raft cache-load-shim pattern); `prefetch.global.L1` -> no-op on HIP; `asm("trap;")` -> `__builtin_trap()`/`abort()` (and revisit TTileReducer<64> since a 64-wide tile is natural on wave64). USE_HIP-guard each; CUDA keeps PTX.
6. **`__CUDA_ARCH__` guards take the host/fallback branch on HIP (MED).** ~125 numeric `#if __CUDA_ARCH__ <op> NNN` sites (mostly `<600`, `<=350`, `>=520`, `<700`). `__CUDA_ARCH__` is UNDEFINED in HIP device compilation (cudaKDTree/MPPI/raft family). Most select an old-arch portable fallback (safe on HIP) vs a newer intrinsic; the fallback is generally what we want. BUT: (a) the double-atomicAdd `<600` CAS fallback is taken on HIP -- functionally correct, just slower than native FP64 atomicAdd (gfx90a has it; consider a USE_HIP fast path); (b) `atomicAdd_block` under `>=600` is skipped on HIP -> falls back to global atomicAdd (correct). Decide per-site; do NOT blanket-`#define __CUDA_ARCH__` (cudf cascade lesson). fp16 is only in dcg.cu so no half-intrinsic activation cascade.
7. **Rule-of-five on graph/event/stream RAII handles (MED).** `TCudaGraph`/`TCudaEvent`/stream wrappers ref-count an inner handle and call hipGraphDestroy/hipEventDestroy in the destructor; AMD faults on destroying a default/double-destroyed handle where CUDA tolerates it (colmap CuTexObj lesson). Audit default-constructed inner handles (`cudaGraph_t Graph_` uninit) and guard destructors (`if (handle) hipXDestroy(handle)`).
8. **CUDA graph / stream-capture semantics (MED).** HIP graphs + stream capture exist but have had robustness/coverage deltas (cf. rmm hipStreamSynchronize-after-destroy tolerance; MPPI hipFFT). The engine drives training through captured graphs -- exercise an actual GPU training run, not just unit tests, and watch for capture-mode or graph-instantiate differences.
9. **ya-make-generated CMake (PROCESS risk, not a fault class).** Edits to `CMakeLists*.txt` are "supposed" to be ported back to ya.make by maintainers. Keep our CMake changes additive/simple (USE_HIP option, sibling HIP source-list reuse, function override) so an upstream PR is reviewable; the upstream-acceptance path will need maintainer discussion (the PR is gated behind user approval anyway).
10. **Coarse-grained managed-memory atomics (LOW).** cudaMallocManaged + atomics, but only Add/CAS/Or are used (the cudaKDTree int atomicMin/Max drop does not apply). Low risk; verify no surprising RMW on managed buffers.
11. **`__syncwarp()` on wave64 (LOW-MED).** Numerous `__syncwarp()` calls assume a 32-lane warp; on HIP `__syncwarp()` exists but syncs the (64-lane) wavefront. Combined with risk 1, prefer removing the warp-synchronous tails over relying on `__syncwarp` width.

## Build commands (gfx90a)
Dependencies: ROCm 7.2.1 (hipcc, rocThrust, hipCUB, rocPRIM -- all in `/opt/rocm`). No conda/MOAT deps. The host build needs the Yandex monorepo slice that ships in-tree (util/, contrib/, library/) plus standard host deps (the CUDA-less host build already works via conan/CMake).

Direct CMake (preferred for iteration):
```
cmake -G Ninja -S projects/catboost/src -B projects/catboost/src/build_hip \
  -DCMAKE_BUILD_TYPE=Release \
  -DHAVE_HIP=ON -DUSE_HIP=ON \
  -DCMAKE_HIP_ARCHITECTURES=gfx90a \
  -DCMAKE_HIP_COMPILER=/opt/rocm/llvm/bin/clang++ \
  -DCMAKE_PREFIX_PATH=/opt/rocm
cmake --build projects/catboost/src/build_hip --target catboost-cuda-cuda_util-ut catboost
```
(The exact `-DHAVE_HIP`/`-DUSE_HIP` option names are introduced by this port; mirror the existing HAVE_CUDA plumbing.) The same commit builds gfx1100/gfx1151 by changing only `-DCMAKE_HIP_ARCHITECTURES`.

build_native.py path (once the option is wired): extend it with `--have-hip`/`--rocm-root-dir` analogous to `--have-cuda`/`--cuda-root-dir` (build/build_native.py:105-107,484-487).

CPU-only ROCm compile smoketest (manual, not a gate): `rocm/dev-ubuntu-24.04:7.2.4-complete`.

## Test plan
GPU correctness (the validator's real gate) -- the project's own C++ unit tests under `catboost/cuda/*/ut/`, built as Yandex `unittest_main` executables (`add_yunittest`), run on the assigned gfx90a:
- `catboost-cuda-cuda_util-ut`: test_reduce, test_scan, test_segmented_scan, test_segmented_sort, test_sort, test_fill, test_transform, test_reorder_and_partition, test_compression_gpu, test_calculate_threshold. These directly exercise the wave64-sensitive reduce/scan/sort primitives -- the highest-signal correctness tests.
- `catboost-cuda-cuda_lib-ut`: GPU runtime/memory/stream/graph layer.
- `catboost-cuda-gpu_data-ut`, `catboost-cuda-methods-ut`, `catboost-cuda-ctrs-ut`, `catboost-cuda-targets-ut`, `catboost-cuda-train_lib-ut`: histogram/method/target kernels and end-to-end GPU training pieces.
Run SERIALLY on a single assigned GPU (ctest without -jN; MPPI lesson: parallel test procs on one GPU cause false transient failures). Run each twice with a fixed seed and assert bit-identical where the algorithm is deterministic (reduce/scan/sort) -- the determinism check is the fingerprint for residual wave64 reduction races.

End-to-end GPU training validation (proves the captured-graph training path, not just primitives): train a small CatBoost model with `task_type=GPU` on a bundled/standard dataset via the Python package (or the `catboost` CLI built with HAVE_HIP) and compare metrics/predictions against the CPU path within tolerance, plus run-to-run stability. GBDT GPU training is not bitwise-reproducible vs CPU (different reduction order) but should match metrics closely and be stable across runs.

Python GPU tests: `catboost/pytest/` contains the training test suite; the GPU-marked subset (task_type=GPU) is the integration gate once the `_catboost` shared lib builds with HIP.

Non-GPU regression set (must NOT regress): the entire CPU build + CPU unit tests are untouched by a clean Strategy-A port (host C++ sees no HIP). Confirm a HAVE_CUDA=OFF/HAVE_HIP=OFF build still configures and that CPU `ut` targets pass. The NVIDIA CUDA path must remain byte-identical (all HIP changes are USE_HIP-guarded or in the new cuda.cmake HIP branch).

## Open questions
1. Upstream-acceptance: given the "CMake is generated from ya.make; only simple edits accepted" policy, will maintainers take a HIP build via the generated CMake, or must it go through ya.make? The MOAT fork builds regardless; flag for the user before the upstream PR. Keep CMake edits minimal/additive to maximize reviewability.
2. Histogram-kernel wave64 strategy per kernel (two-32-lane-halves vs native-64) -- decide during porting by how each kernel's partials recombine (positional shared-mem layout -> halves; atomicAdd-combined -> native 64). Bitonic/radix-sort-in-shared paths (BlockRadixSort) need the wave64 sort-correctness check.
3. Whether gfx90a's native FP64 atomicAdd should replace the `__CUDA_ARCH__<600` CAS fallback for perf (correctness is fine either way).
4. Multi-GPU/MPI path (`cuda_lib/mpi/`) -- scope it out of the initial single-GPU validation unless trivially building.
