# LMCache -- ROCm/HIP port plan (lead platform: linux-gfx90a)

## Project
- Name: LMCache
- Upstream: https://github.com/LMCache/LMCache
- Default branch: dev
- Pristine clone HEAD analyzed: 7eb57ad ("[Perf] Avoid redundant PCIe transfer on leader rank during retrieve", depth-1 of dev)
- What it is: a KV-cache offload/reuse layer for LLM serving (vLLM / SGLang / TRT-LLM engine extension). The CUDA surface being ported is the `lmcache.c_ops` PyTorch extension in `csrc/`: KV-cache layout-transform kernels, an arithmetic-coding (CacheGen) encoder/decoder, a CDF kernel, and a fused rotary-embedding kernel, plus host-side pinned/NUMA/hugepage allocators and a CUDA host-callback event recorder.

## Existing AMD support (finding + decision)
Classification: improvable / finish-the-rot (NOT already-supported, NOT abandoned).

LMCache already INTENDS to support ROCm and ships most of the scaffolding:
- `setup.py` has a `BUILD_WITH_HIP=1` branch (`rocm_extension()`) that calls a `hipify_wrapper()` (torch hipify, `is_pytorch_extension=True`) and lists hipified source names (`csrc/pybind_hip.cpp`, `csrc/mem_kernels.hip`, `csrc/mem_alloc_hip.cpp`, `csrc/utils_hip.cpp`, ...).
- Source-level `#ifdef USE_ROCM` guards already exist in 6 files: `csrc/cuda_compat.h` (LMCACHE_LDG = `*(arg)` vs `__ldg`), `csrc/mem_kernels.cu` (`<hip/hip_fp8.h>` vs `<cuda_fp8.h>`), `csrc/event_recorder.{cpp,h}` (drops `CUDART_CB`, `hipLaunchHostFunc` vs `cudaLaunchHostFunc`), `csrc/completion_recorder.{cpp,h}`.
- `docker/Dockerfile.rocm` + `docker/Dockerfile.rocm-lightweight` build with `BUILD_WITH_HIP=1 CXX=hipcc` against vLLM-ROCm wheels, `requirements/rocm_core.txt` (cupy-rocm-7-0), `.buildkite/scripts/pick-free-gpu-amd.sh`, and a 1520-line `lmcache/python_ops_fallback.py` pure-Python reference used by the parity tests.

Why this is still a real MOAT target (per PORTING_GUIDE "assess existing AMD support" + the GPUMD lesson, 2026-05-30 changelog: "a project may already ship an upstream HIP path ... do NOT auto-skip as already-supported. These HIP makefiles bitrot against the ROCm in use; the MOAT value is to actually build+GPU-validate them and fix the rot"):
1. Upstream targets only CDNA3/CDNA4 (`Dockerfile.rocm` PYTORCH_ROCM_ARCH="gfx942,gfx950" = MI300X/MI350X). **gfx90a (MI250X, CDNA2) is NOT a target they build or test.** No evidence the HIP path has ever been exercised on wave64-CDNA2 / ROCm 7.2.1.
2. The HIP build path is plausibly never CI-tested at all on real GPU (CI is NVIDIA; the ROCm Dockerfiles are build-only recipes). There is no GPU-validated proof it runs correctly.

Decision: PROCEED. Finish + GPU-validate the existing HIP path on gfx90a (ROCm 7.2.1). Expect the work to be: (a) a working build via the upstream `BUILD_WITH_HIP=1` flow, (b) add/confirm gfx90a in the arch surface, (c) fix whatever wave64 / ROCm-7.2.1-runtime faults surface at test time. The footprint should be small; this is correctness-first, not a rewrite.

## Build classification (evidence)
torch-extension (Strategy B). Evidence:
- `setup.py` `cuda_extension()` builds `torch.utils.cpp_extension.CUDAExtension("lmcache.c_ops", ...)` with `cmdclass={"build_ext": cpp_extension.BuildExtension}` (setup.py lines ~217-252).
- Kernels include `<torch/all.h>`, `<torch/extension.h>`, `<ATen/cuda/CUDAContext.h>`, `<c10/cuda/CUDAGuard.h>`, use `AT_DISPATCH_*` (via `csrc/dispatch_utils.h`) and `pybind11` (`csrc/pybind.cpp` -> `PYBIND11_MODULE(c_ops, ...)`).
- Root `CMakeLists.txt` is a 2-line stub (`project(lmcache LANGUAGES CXX)`); the GPU build is entirely setup.py-driven. So this is NOT a pure-CMake (Strategy A) project.
- ext_type set to `torch-extension` in upstream.json + status.json.

## Port strategy (B: torch-hipify) + rationale
Use the project's own `BUILD_WITH_HIP=1` flow; do NOT hand-add a compat header or rename symbols (Strategy B rule). torch hipify rewrites the `.cu`/`.cpp`/`.cuh` sources at build time.

Confirmed hipify behavior on THIS host (torch 2.13.0a0+gitb5e90ff, HIP 7.2.53211), by running `csrc/`'s exact `hipify_wrapper()` on a throwaway copy:
- With `is_pytorch_extension=True`, hipify writes the hipified files IN PLACE into `csrc/` (the `output_directory=csrc_hip/` arg only receives verbatim copies of the originals; it is effectively ignored for the rewrite). Naming: `*.cu -> *.hip`, `pybind.cpp -> pybind_hip.cpp`, `mem_alloc.cpp -> mem_alloc_hip.cpp`, `utils.cpp -> utils_hip.cpp`, `*.cuh -> *_hip.cuh`, `event_recorder.* -> event_recorder_hip.*`, `completion_recorder.* -> completion_recorder_hip.*`.
- This EXACTLY matches the source list in `rocm_extension()`. So the apparent "rocm_extension references files that do not exist in the pristine tree" is NOT a bug: those files are generated by `hipify_wrapper()` at build time. The setup.py ROCm glue is correct as written.
- The hipified output is clean: `c10/cuda/CUDAGuard.h -> c10/hip/HIPGuard.h`, `cudaError_t/cudaSuccess/cudaMemcpyAsync/cudaHostRegister/cudaHostGetDevicePointer -> hip*`, `<<<>>> -> hipLaunchKernelGGL`, includes rewritten to the `*_hip.cuh` names, and ZERO unmapped `cuda[A-Z]*` symbols left in the `.hip` files. The PyTorch C++ API names `at::cuda::OptionalCUDAGuard` / `at::cuda::getCurrentCUDAStream()` are intentionally left as-is (they exist on ROCm torch; they are torch symbols, not CUDA-toolkit symbols).

Implication: the porter's job is mostly (1) drive the build green on gfx90a and (2) GPU-validate + fix runtime faults, not symbol translation. Most likely the only source edits (if any) are wave64-correctness fixes the validator surfaces, plus a small arch/docs touch to include gfx90a.

## CUDA surface inventory
All GPU code is in `csrc/`. ~8.2k lines total; the kernels:
- `csrc/mem_kernels.cu` (1089 L): KV-cache layout transforms (multi_layer / single_layer / sgl / flash / block transfer) between LMCache paged buffers and engine KV caches. Grid-stride copy kernels; `__host__ __device__` layout helpers; `cudaMemcpyAsync`, `cudaHostGetDevicePointer`, `at::cuda::OptionalCUDAGuard`, `getCurrentCUDAStream`. Includes `<hip/hip_fp8.h>` under USE_ROCM (fp8 header is included but the fp8 TYPE is not actually referenced by name; harmless).
- `csrc/mp_mem_kernels.cu` (390 L): multiprocess / block KV transfer kernels (grid-stride, `threadIdx`/`blockIdx` indexing). No warp ops.
- `csrc/ac_enc.cu` (476 L) + `csrc/ac_dec.cu` (538 L): CacheGen arithmetic coder. PER-THREAD (per-channel) sequential range coder; `__shared__` CDF + bytestream staging, `__syncthreads()` block coordination, software `binsearch`. No warp primitives, no atomics. Deterministic by construction (each thread owns one channel).
- `csrc/cal_cdf.cu` (129 L): per-channel histogram + prefix-sum + CDF normalization. `__shared__ uint16_t hist[64][BLOCK_SIZE]` (each thread owns its own `[*][threadIdx.x]` column; no cross-lane sharing), `__syncthreads()`. BLOCK_SIZE in {1..128}.
- `csrc/pos_kernels.cu` (148 L): fused rotary-embedding K kernel (vLLM-derived); one block per token, grid-stride over heads*dim. No warp ops.
- Kernel headers: `csrc/*.cuh`, `csrc/cuda_compat.h` (LMCACHE_LDG), `csrc/dispatch_utils.h` (AT_DISPATCH wrappers).
- Host C++: `csrc/mem_alloc.cpp` (235 L; `cudaHostAlloc`/`cudaFreeHost`/`cudaHostRegister`/`cudaHostUnregister` over pinned + NUMA (`set_mempolicy`/`mmap`) + 2MB-hugepage + shm memory), `csrc/event_recorder.cpp` (`cudaLaunchHostFunc` host callback to stamp wall-clock on stream events), `csrc/completion_recorder.cpp` (same pattern), `csrc/utils.cpp` (`get_gpu_pci_bus_id`), `csrc/pybind.cpp` (module def).

Library / intrinsic usage (full grep of csrc):
- Warp intrinsics: NONE. No `__shfl*`, `__ballot`, `__activemask`, `__any/__all`, `__popc`, `warpSize`, `__syncwarp`, no `0xffffffff` masks, no literal-32 warp assumptions. (This is the single most important inventory result -- see Risk list.)
- Atomics: NONE.
- Textures / surfaces: NONE.
- Cooperative groups: NONE.
- cuBLAS / cuFFT / cuRAND / cuSPARSE / cuSOLVER / cuDNN / Thrust / CUB / NCCL: NONE.
- half/bf16 device intrinsics (`__hadd`, `__float2half`, `__nv_*`): NONE (dtype dispatch is via AT_DISPATCH Float/Half/BFloat16; arithmetic is plain `scalar_t`).
- Streams/events: `at::cuda::getCurrentCUDAStream()`, `at::cuda::OptionalCUDAGuard`, `cudaLaunchHostFunc`/`hipLaunchHostFunc` (guarded), `cudaMemcpyAsync` -- all torch/HIP 1:1.
- Pinned/managed: pinned host alloc + host-register (`cudaHostAlloc`/`cudaHostRegister`), incl. over hugetlb + NUMA + shm; `cudaHostGetDevicePointer`. No `cudaMallocManaged`.

ROCm/HIP mapping: every symbol above is a torch-hipify 1:1 mapping (verified empirically). No library swaps required (no cuBLAS/cuFFT/etc.). Expected mechanical port.

## Risk list (expected fault classes, with PORTING_GUIDE citations)
Ordered by likelihood/impact. This codebase is unusually low-risk for a Strategy-B port because it has no warp/atomic/texture/library surface.

1. wave64 vs wave32 -- LOW/likely-NONE (Fault classes: "Warp size"). There are zero warp intrinsics and zero hardcoded-32 warp assumptions in the entire CUDA surface, so the canonical wave-size class (PORTING_GUIDE lines 86-103, 141, 143, 157, 186, 200, 218) almost certainly does not apply. All cross-thread coordination is block-wide `__syncthreads()`, which is wave-size agnostic. The `BLOCK_SIZE` template constants in `cal_cdf.cu`/`ac_*.cu` are launch-tiling sizes (blockDim.x), not warp widths; each thread is fully independent within the block. Validator should still confirm by (a) empty warp-primitive grep on the hipified `.hip` (already true on `.cu`) and (b) determinism + correctness of the arithmetic coder and CDF kernel on gfx90a, not by the presence of `32` literals (cf. fused-ssim lesson, line 184).

2. CDNA2 64KB/block shared-memory cap -- LOW (PORTING_GUIDE line 221: a `* 1024` / fixed-byte shared size >64KB silently fails on gfx90a). Measured the static `__shared__` footprints:
   - `ac_dec.cu` / `ac_enc.cu`: budget is explicitly capped by `MAX_SHARED_MEMORY_PER_THREAD = 0xc000 / MAX_THREAD_PER_BLOCK` = 49152/128 = 384 B/thread; total static shared <= 0xc000 = 48 KB. Under 64 KB. OK.
   - `cal_cdf.cu`: `hist[64][BLOCK_SIZE]` uint16 with BLOCK_SIZE<=128 => 64*128*2 = 16 KB. OK.
   None use `cudaFuncAttributeMaxDynamicSharedMemorySize` opt-in, so no >64KB request exists. Low risk, but the validator should still watch for an "invalid argument" at the cachegen kernel launches (the sticky-error misattribution trap, line 221) if any config pushes shared higher.

3. Host pinned/hugepage/NUMA register on ROCm -- MEDIUM robustness risk (not a port-strategy blocker; Fault classes "Library swaps" is N/A, this is a runtime-behavior diff). `csrc/mem_alloc.cpp` calls `cudaHostRegister` on MAP_HUGETLB (2MB hugepage) and NUMA-bound `mmap` regions, and `cudaHostGetDevicePointer` in `mem_kernels.cu`. `hipHostRegister`/`hipHostGetDevicePointer` exist and map 1:1, but ROCm may differ from CUDA in (a) whether registering hugetlb-backed memory succeeds, (b) `hipHostGetDevicePointer` requiring the `hipHostRegisterMapped` flag. Same family as the rmm/MPPI "HIP-vs-CUDA robustness diffs" (lines 211, 164). Validator must exercise `tests/v1/test_memory_management.py` + the pinned-allocator paths and, if hugepage/NUMA register fails on ROCm, decide between a USE_ROCM flag fix and scoping that allocator variant out with documentation. This is the most likely place to need an actual code touch.

4. CacheGen arithmetic-coder numeric exactness on gfx90a -- LOW-MEDIUM (no specific guide line; general "AMD strict where CUDA lenient" + determinism bar from lines 157, 165-166, 174). The coder uses 64-bit intermediate arithmetic (`uint64_t span`, `* c_count`, `>> precision`) and integer ops only -- no FP, no fast-math reassociation -- so encode/decode should be bit-exact and deterministic across vendors. Risk is only if some integer-overflow/UB edge that nvcc tolerates is miscompiled by clang/HIP. Validate by encode->decode round-trip exactness AND run-to-run determinism on real KV data (tests/benchmarks/test_cachegen.py).

5. fp8 header include -- LOW (Library swaps). `csrc/mem_kernels.cu` includes `<hip/hip_fp8.h>` under USE_ROCM; the fp8 type is not used by name, so this is just a header presence check on ROCm 7.2.1. If `<hip/hip_fp8.h>` is absent/renamed it is a trivial fix (drop the include or guard it), not a kernel rewrite.

6. arch surface omits gfx90a -- LOW build-config item (PORTING_GUIDE line 149: configurable arch, no hardcoded literal). Upstream's ROCm path reads `PYTORCH_ROCM_ARCH` (torch's ROCm arch env). For the lead bringup, build with `PYTORCH_ROCM_ARCH=gfx90a` (or the full `gfx90a;gfx1100;gfx1151` set so followers need no rebuild). If any fork-side arch edit is needed, keep it env-driven, never a literal -- so gfx1100/gfx1151 followers need no source change (avoids head_sha churn / forced revalidation).

NOT-applicable fault classes (ruled out by inventory, noted so the porter does not chase them): texture pitch / linear-filter / layered-array (no textures); rule-of-five on resource handles (no raw stream/event/texture handle wrappers -- streams come from torch); OOB neighbor reads (no stencil/neighbor kernels); warp-synchronous reductions (no warp reductions); cuBLAS/cuFFT/cuSPARSE/cuSOLVER swaps (none used); sub-word atomic CAS (no atomics); CUTLASS/CuTe (none); cuda::std / libcudacxx (none; plain C++17 + ATen).

## File-by-file change list (expected; minimal-footprint)
The default expectation is ZERO source edits to the kernels (torch hipify handles them) beyond what validation surfaces. Concretely:
- `setup.py` -- likely UNCHANGED. The `rocm_extension()` + `hipify_wrapper()` flow already produces and consumes the correct in-place hipified filenames (verified empirically). Touch only if a real glitch appears (e.g. add gfx90a to an arch default, or pass `--offload-arch` if `PYTORCH_ROCM_ARCH` is not honored by the installed torch).
- `csrc/mem_alloc.cpp` -- POSSIBLE small USE_ROCM touch if `hipHostRegister` on hugetlb/NUMA or `hipHostGetDevicePointer` mapping differs from CUDA (Risk 3). Determined by `test_memory_management.py` on gfx90a.
- `csrc/mem_kernels.cu` -- POSSIBLE trivial touch to the `<hip/hip_fp8.h>` include only if that header is unavailable on ROCm 7.2.1 (Risk 5).
- Any kernel `.cu` -- ONLY if a wave64-correctness or clang/HIP-compile fault surfaces at validation (not expected given the inventory). Guard any such fix with `USE_ROCM` so the NVIDIA build is byte-identical.
- `docker/Dockerfile.rocm` / docs -- OPTIONAL: extend the documented arch list to include gfx90a (MI250) so the port is reproducible. Treat as nice-to-have; weigh against head_sha churn for followers (PORTING_GUIDE line 119) -- prefer env-driven arch over editing committed files.
- `projects/LMCache/notes.md` (MOAT side, not the fork) -- record the build script, the hipify-in-place behavior, the exact GPU test commands, and any scoped-out tests.

## Build commands (gfx90a)
Host already has a ROCm PyTorch (torch 2.13.0a0+gitb5e90ff, HIP 7.2.53211) -- build the extension against it with the project's own HIP flow. From the fork clone root:

    # one-time build deps (per requirements/build.txt)
    pip install ninja "packaging>=24.2" "setuptools>=77.0.3,<81.0.0" "setuptools_scm>=8" wheel

    # build the c_ops HIP extension in-place against the installed ROCm torch
    BUILD_WITH_HIP=1 CXX=hipcc PYTORCH_ROCM_ARCH=gfx90a \
      python3 setup.py build_ext --inplace --no-build-isolation
    # or: BUILD_WITH_HIP=1 CXX=hipcc PYTORCH_ROCM_ARCH=gfx90a \
    #       pip install -e . --no-build-isolation -v

Notes:
- `hipify_wrapper()` runs first and writes `csrc/*_hip.*` + `csrc/*.hip` in place; if a `.cu`/`.cpp` is later edited, re-run the build (it re-hipifies) -- the Strategy-B stale-mirror gotcha (PORTING_GUIDE line 79). A clean re-hipify = `rm -f csrc/*.hip csrc/*_hip.* && rm -rf csrc_hip` before rebuild.
- To bring up all AMD targets at once (avoid follower rebuild churn): `PYTORCH_ROCM_ARCH="gfx90a;gfx1100;gfx1151"`.
- A CPU-only `rocm/dev-ubuntu-24.04:7.2.4-complete` docker build is a compile smoketest only, never the validation gate (PORTING_GUIDE line 120; CLAUDE.md Testing).
- Disable Actions on the jeffdaily fork after forking (CLAUDE.md): `gh api -X PUT repos/jeffdaily/LMCache/actions/permissions -F enabled=false`.

## Test plan
GPU correctness is gated by `torch.cuda.is_available()` (True on ROCm torch) and `device="cuda"` (maps to the AMD GPU under ROCm). The c_ops tests import `lmcache.c_ops` and validate the kernels against torch reference and the pure-Python `lmcache/python_ops_fallback.py`.

Primary GPU tests (the validator MUST run these on gfx90a):
- `tests/v1/test_mem_kernels.py` -- KV-cache layout-transform kernels (multi_layer / single_layer / sgl / flash), H2D/D2H round-trips across all GPUKVFormat layouts. The core of `mem_kernels.cu`.
- `tests/v1/test_mp_mem_kernels.py` -- multiprocess / block-transfer kernels (`mp_mem_kernels.cu`).
- `tests/v1/test_pos_kernels.py` -- fused rotary embedding (`pos_kernels.cu`); compare against a torch/vLLM reference.
- `tests/benchmarks/test_cachegen.py` -- CacheGen arithmetic-coding encode->decode round-trip + CDF (`ac_enc.cu`, `ac_dec.cu`, `cal_cdf.cu`). Assert exact decode recovery AND run-to-run determinism (the integer coder must be bit-exact, lines 157/174).
- `tests/v1/test_c_ops_fallback_parity.py` + `tests/v1/test_python_ops_fallback.py` -- c_ops vs pure-Python parity (signatures + numeric results).
- `tests/v1/test_gpu_connector.py` -- end-to-end GPU connector exercising the kernels through the public API.
- `tests/v1/test_memory_management.py` -- pinned/NUMA/hugepage allocator paths (`mem_alloc.cpp`); the most likely place to surface a HIP host-register robustness diff (Risk 3).
- `tests/v1/mp_observability/test_event_recorder.py` + `tests/v1/multiprocess/test_completion_recorder.py` -- the `cudaLaunchHostFunc`/`hipLaunchHostFunc` host-callback path (`event_recorder.cpp`, `completion_recorder.cpp`).

Run command (single assigned GPU -> serial, per PORTING_GUIDE line 165):
    HIP_VISIBLE_DEVICES=<id> python3 -m pytest -p no:cacheprovider \
      tests/v1/test_mem_kernels.py tests/v1/test_mp_mem_kernels.py \
      tests/v1/test_pos_kernels.py tests/v1/test_c_ops_fallback_parity.py \
      tests/v1/test_python_ops_fallback.py tests/v1/test_gpu_connector.py \
      tests/v1/test_memory_management.py tests/benchmarks/test_cachegen.py \
      tests/v1/mp_observability/test_event_recorder.py \
      tests/v1/multiprocess/test_completion_recorder.py -v

Determinism check for the arithmetic coder: run test_cachegen.py twice and assert the encoded bytestreams + decoded tensors are bit-identical across runs (integer-only coder; any nondeterminism is a real bug).

Non-GPU regression set (must not regress; runnable without a GPU): the pure-CPP storage extensions and Python suites that do NOT require c_ops -- e.g. `tests/test_serde.py`, `tests/test_utils.py`, `tests/v1/test_config.py`, `tests/v1/test_token_database.py`, `tests/v1/test_cache_policy.py`, `tests/v1/storage_backend/`, `tests/cli/`. These exercise `native_storage_ops` / `lmcache_redis` / `lmcache_fs` (built in every mode) and pure-Python logic; the HIP build must not break them. Note: many integration tests need vLLM/SGLang/redis and are out of scope for a kernel port -- run the subset that imports cleanly.

GPU env required: a ROCm PyTorch (present on this host). vLLM-ROCm is needed only for the engine-integration tests, not the kernel/parity tests above.

## Inter-project MOAT deps
NONE. The `lmcache.c_ops` GPU extension has no build dependency on any other MOAT-ported CUDA library:
- No cuBLAS/cuFFT/cuSPARSE/Thrust/CUB/CUTLASS/cuCollections/rmm/raft, no submodules, no FetchContent/find_package of a CUDA lib in `csrc/` or setup.py.

## Open questions
