# marian-dev ROCm/HIP port plan (lead platform: linux-gfx90a)

## Project
- Name: marian-dev
- Upstream: https://github.com/marian-nmt/marian-dev (marian-nmt/marian-dev)
- Default branch: master
- Head at clone: c9f287d ("Merged PR 39147: Fix LSH memory initialization for dynamic invocation")
- Marian is a standalone C++ neural machine translation toolkit (trainer + decoder + scorer). GPU backend is custom CUDA kernels + cuBLAS/cuBLASLt/cuSPARSE/cuRAND, NCCL for multi-GPU, cuDNN only for an optional (off-by-default) convolution/pooling path.

## Existing AMD support
- Finding: NONE. No USE_HIP, no __HIP_PLATFORM, no hipify scaffolding, no makefile.hip, no stale ROCm branch/PR anywhere in the tree. The only GPU backend is CUDA. There is no OpenCL/Vulkan/SYCL path either.
- Decision: PROCEED with a fresh CUDA-to-HIP port. This adds real value (first AMD GPU support for Marian).

## Build classification
- cmake (pure CMake, NOT a pytorch extension). Evidence:
  - Top-level CMakeLists.txt:386 `find_package(CUDA "9.0")`, no `find_package(Torch)`, no torch/setup.py/pyproject.
  - GPU TUs are compiled via the LEGACY FindCUDA module: src/CMakeLists.txt:225 `cuda_add_library(marian_cuda ... STATIC)` over the 16 .cu/.cpp GPU sources, driving `CUDA_NVCC_FLAGS` (CMakeLists.txt:548-555).
  - `option(COMPILE_CUDA ... ON)` (CMakeLists.txt:22) gates the whole GPU build.
- This is the legacy `cuda_add_library` variant of Strategy A (cupoch fault class), NOT the modern `set_source_files_properties(.. LANGUAGE HIP)` form.

## Port strategy: A (compat-header + cuda_add_library shim), correctness-first mechanical port
Rationale: small .cu surface (16 GPU TUs, ~one large kernel file), libraries all have ROCm equivalents (hipBLAS/hipBLASLt/hipSPARSE/hipRAND/rocThrust present in ROCm 7.2.1), no CUTLASS/CuTe/wgmma/tensor-core code. A mechanical HIP port is the right first step; no AMD-native rewrite is warranted for classic Marian (it relies on vendor BLAS for GEMM, not hand-tuned MMA kernels). Concretely:
1. Add `src/tensors/gpu/cuda_to_hip.h` compat header. On HIP it `#include <hip/hip_runtime.h>` and aliases the cuda* runtime symbols Marian uses (full list in CUDA surface below) to hip*; on NVIDIA it is a no-op `#include <cuda_runtime.h>`. This is the only file that knows HIP.
2. FP16/__CUDACC__ steering (MPPI-Generic + 3DUNDERWORLD lessons): src/common/types.h and src/functional/operators.h gate the device-half path and toolkit includes on `#ifdef __CUDACC__` (hipcc does NOT define it) and then `#include <cuda.h>`/`<cuda_fp16.h>`. In the compat header, AFTER `<hip/hip_runtime.h>` is parsed, define `__CUDACC__` when `__HIPCC__` is set so the CUDA half path is taken, and provide forwarding shim headers (named `cuda.h`, `cuda_fp16.h`, `cuda_runtime.h`, `cublas_v2.h`, `cublasLt.h`, `cusparse.h`, `cusparse_v2.h`, `curand.h`, `cooperative_groups.h`) in a `hip_compat/` dir on the HIP-only include path (each forwards to the HIP equivalent: cuda_fp16.h -> `<hip/hip_fp16.h>`, cooperative_groups -> `<hip/hip_cooperative_groups.h>`, cublasLt.h -> `<hipblaslt/hipblaslt.h>`, etc.). Force-include the compat header on every HIP TU via `CMAKE_HIP_FLAGS -include .../cuda_to_hip.h` so the defines/aliases precede any use regardless of per-file include order. On NVIDIA the shim dir is absent so the real toolkit headers win (byte-identical CUDA path). Include `<cstring>`/`<cstdlib>` before hip_runtime in the compat header (gpuRIR host-memcpy-in-.cu trap).
3. CMake: add `option(USE_HIP ... OFF)`. Under USE_HIP, `enable_language(HIP)`, define a `cuda_add_library()` SHIM macro (cupoch lesson) that calls `add_library()` + marks the .cu (and the mixed-in prod.cpp/prod_sparse.cpp) `LANGUAGE HIP` + sets `HIP_ARCHITECTURES "${CMAKE_HIP_ARCHITECTURES}"` (default gfx90a only when unset -- never a literal, so followers need no source edit). Map `CUDA_LIBS`/`CUDA_*_LIBRARY` -> hip:: equivalents (hip::host for runtime; link hipblas, hipblaslt, hipsparse, hiprand; do NOT link roc::rocthrust/hip::device PUBLIC -- they inject `-x hip`/`--offload-arch` into host .cpp; rocThrust/hipCUB are header-only on the HIP default include path, cupoch/STRUMPACK lesson). Translate `CUDA_NVCC_FLAGS` (--extended-lambda, --use_fast_math, -gencode...) to HIP-appropriate flags; drop the -gencode COMPUTE list (HIP_ARCHITECTURES drives --offload-arch). Pin `-ffp-contract=on` and consider `--offload-compress` if link size warrants (single arch should be fine).
4. Guard genuinely divergent code with `#if defined(USE_HIP)`; keep guards rare.

## Dependency contract (the make-or-break verdict)
- cuDNN -> MIOpen: NOT REQUIRED for the core port. `option(USE_CUDNN ... OFF)` (CMakeLists.txt:32) is OFF by default. ALL cuDNN code is behind `#ifdef CUDNN`: the entire body of src/tensors/gpu/cudnn_wrappers.cu, plus convolution/pooling node operators (graph/node_operators_binary.h:10/1734, layers/convolution.cpp:6). With USE_CUDNN=OFF the cudnn TU is empty and the conv/pool ops are not built. Convolution/pooling is a legacy char-CNN feature; the default Transformer/RNN NMT path never touches it. VERDICT: leave the CUDNN path CUDA-only (do not port to MIOpen in this pass); build and validate with USE_CUDNN=OFF. A MIOpen port is a separable, optional follow-up, NOT a blocker. This is the single most important tractability finding -- cuDNN does NOT make this a blocked effort.
- cuBLAS -> hipBLAS (present /opt/rocm). cublasSgemm/GemmEx/GemmBatchedEx/SetMathMode -> hipblas* (1:1; watch hipBLAS v2 compute-type enums, faiss MatrixMult version-guard pattern).
- cuBLASLt -> hipBLASLt (present, libhipblaslt.so.1.2.70201). The affine+bias+ReLU epilogue (prod.cpp:577-696, `#if CUDA_VERSION >= 11000`) maps to hipblaslt: HIPBLASLT_EPILOGUE_{BIAS,RELU_BIAS,DEFAULT}, MATMUL_DESC_{TRANSA,TRANSB,EPILOGUE,BIAS_POINTER}, PREF_MAX_WORKSPACE_BYTES, HIPBLAS_COMPUTE_{32F,16F} all exist. RISK (confirmed): hipBLASLt has NO `MATMUL_PREF_MIN_ALIGNMENT_{A,B,C,D}_BYTES` preference attrs (only MAX_WORKSPACE_BYTES and SEARCH_MODE) -- the five SetAttribute alignment calls at prod.cpp:620-623 must be dropped/USE_HIP-guarded. Also CUBLAS_COMPUTE_32F_FAST_16F may have no hipblaslt analog -> map to HIPBLAS_COMPUTE_32F.
- cuSPARSE -> hipSPARSE (present). Marian carries BOTH a legacy path (prod_sparse_cu10.h: cusparseScsrmm/Sgemmi, `CUDA_VERSION < 11000`) and a GENERIC-API path (prod_sparse_cu11.h: cusparseCreateCsr/CreateDnMat/SpMM/SpMM_bufferSize, `CUDA_VERSION >= 11000`). Port the GENERIC path through hipSPARSE name-for-name (amgcl lesson: hipsparseCreateCsr/CreateDnMat/SpMM, HIPSPARSE_*, hipDataType/HIP_R_* from <hip/library_types.h>). Sparse GEMM is used only for sparse-attention/LSH models, not the default dense Transformer.
- cuRAND -> hipRAND (present). rand.cpp: curandCreateGenerator(Host)/SetPseudoRandomGeneratorSeed/GenerateUniform/GenerateNormal/DestroyGenerator -> hiprand* (1:1).
- Thrust/CUB -> rocThrust (header drop-in). cumsum.cu: thrust scan_by_key/max_element/transform_iterator/device_ptr port unchanged EXCEPT `thrust::cuda::par` -> `thrust::hip::par` under USE_HIP (cupoch namespace-delta lesson). No bare cub:: usage.
- NCCL -> RCCL: NOT required for lead validation. USE_NCCL is ON by default but only enables multi-GPU collectives (training/communicator_nccl.h). The vendored marian-nmt/nccl submodule will not build under hipcc. Configure the lead-platform port with `-DUSE_NCCL=OFF` (single-GPU training/decoding validates the full kernel + BLAS surface). RCCL swap (RCCL is API-compatible with NCCL and ships in ROCm) is a separable follow-up for multi-GPU.
- CUTLASS / CuTe / tensor-core / wgmma: NONE. Confirmed -- Marian uses vendor BLAS for all GEMM. No AMD-native rewrite needed.

## CUDA surface inventory
- GPU TUs (16, via cuda_add_library): layers_new/alibi.cu, tensors/gpu/{cumsum,device,hash,algorithm,prod.cpp,prod.cu,prod_sparse.cpp,topk,element,add,add_all,tensor_operators,cudnn_wrappers}.cu, translator/{nth_element,helpers}.cu. Plus host .cpp pulling toolkit headers: tensors/rand.cpp (curand), tensors/gpu/{backend.h,cuda_helpers.h}, prod_sparse_cu10/cu11.h.
- Custom kernels: element-wise tensor ops + softmax/layernorm/gather/scatter/select (tensor_operators.cu, 3645 lines -- the bulk), broadcasting add (add.cu), full reductions (add_all.cu via 3rd_party/reduce_all.h), top-k (topk.cu), beam-search argmax (nth_element.cu), alibi attention slopes (alibi.cu), prefix-scan (cumsum.cu via rocThrust), hashing/LSH (hash.cu).
- Runtime symbols used (all 1:1 hipify): cudaMalloc/Free/FreeHost/HostAlloc(Default), cudaMemcpy(Async)/MemcpyKinds, cudaMemcpyDefault, cudaSetDevice, cudaGetDeviceProperties/DeviceGetAttribute (cudaDevAttrComputeCapabilityMajor/Minor), cudaDeviceSynchronize, cudaStreamSynchronize, cudaStream_t, cudaGetLastError/GetErrorString, cudaDeviceProp, cudaDataType, cudaError_t/cudaSuccess.
- Warp intrinsics: ALMOST NONE. tensor_operators.cu has one `__popc` (a population count on a hash diff word -- works wave-agnostic). No __shfl/__ballot/__activemask/warpSize in the .cu kernels EXCEPT inside 3rd_party/reduce_all.h (cooperative-groups shfl_down, see Risks). topk.cu / nth_element.cu use a warp-synchronous unrolled reduction tail (see Risks). No hardcoded 32 that reads hardware warp width outside those.
- Cooperative groups: only 3rd_party/reduce_all.h (cg::this_thread_block, cg::sync, cg::tiled_partition<32>, tile.shfl_down). NO cg::reduce, NO cg::labeled_partition (so the gsplat cg::reduce gap does not apply); HIP CG provides tiled_partition/shfl_down.
- FP16: pervasive `half`/`__half` device Ops (operators.h:482+), intrinsics hsin/hcos/hlog/hexp/hsqrt/hrint/hfloor/hceil/__habs -- all present in <hip/hip_fp16.h>. Gated by COMPILE_FP16 which is keyed on __CUDACC__ + CUDA_VERSION (see strategy step 2). A fp16 atomicAdd shim (tensor_operators.cu:29, `__CUDA_ARCH__ >= 700`) -- HIP has native fp16 atomicAdd on gfx90a; the __CUDA_ARCH__ guard takes the host/CAS branch on HIP (functionally fine; the CAS fallback works, or guard to use ::atomicAdd on HIP).
- Textures/surfaces: NONE. No tex/surf usage anywhere. (So the 256B-pitch / layered-array / linear-filter fault classes do not apply.)
- Pinned/managed memory: cudaHostAlloc/cudaFreeHost (device.cu) -> hipHostMalloc/hipHostFree. No managed memory.
- Streams/events: cudaStream_t per-thread (--default-stream per-thread). hipEvent timing is nodiscard (amgcl lesson) -- wrap in the error macro if any timing calls warn.

## Risk list
1. WAVE64 warp-synchronous reduction (HIGH, correctness, on the BLEU path). topk.cu and translator/nth_element.cu reduce shared memory with a `for(s=blockDim.x>>1; s>32; s>>=1){... __syncthreads();}` tree, then an UNROLLED tail `UNROLL_MAXARG_LOOP(32/16/8/4/2/1)` with NO sync, on NON-volatile `__shared__` arrays (topk.cu:43-44, 123-128; nth_element.cu:91-96,183-188,255-260). On CDNA wave64 the low 32 lanes of a 64-lane wavefront are NOT lockstep across those unsynced steps, and non-volatile shared can be register-cached -> wrong AND non-deterministic max/argmax. Exactly the MPPI-Generic fault class. FIX (HIP, guarded): drop the warp-synchronous tail; change the tree condition to `s > 0` and let the `__syncthreads()`-synchronized tree run to 1 (same compare order, block-wide barrier, correct on any wave size). nth_element is the beam-search top-k -- a wrong fix silently corrupts translations, so verify with a fixed-seed determinism check (two runs bit-identical) AND a BLEU compare.
2. 3rd_party/reduce_all.h cooperative-groups final-warp reduction (MEDIUM, correctness). `if(cta.thread_rank()<32){ if(blockSize>=64) mySum+=sdata[tid+32]; for(off=tile.size()/2...) mySum+=tile.shfl_down(mySum,off);}` with `tiled_partition<32>`. Per the gsplat lesson tiled_partition<32> stays within 32 lanes on wave64 (width=32), and the `+sdata[tid+32]` folds the upper 32 of a 64-thread block into the lower 32 before the shfl loop, so this is LIKELY wave64-correct -- but it MUST be proven on real gfx90a (a reduction error shows as wrong layernorm/softmax/loss). If shfl_down width misbehaves, fall back to a __syncthreads tree.
3. __CUDACC__ / __CUDA_ARCH__ guards (MEDIUM, compile + silent-correctness). types.h/operators.h gate the entire device-half path AND toolkit includes on `#ifdef __CUDACC__`; tensor_operators.cu uses `#if __CUDA_ARCH__ >= 700/600`. hipcc defines neither. Define __CUDACC__ (compat header) to take the half path; for device-vs-host data selection use `#if defined(__CUDA_ARCH__) || defined(__HIP_DEVICE_COMPILE__)`. Watch the EIGEN_NO_CUDA interaction only if Eigen appears in device code (it does not here, but defining __CUDACC__ globally can steer other 3rd_party headers -- scope the define carefully).
4. hipBLASLt preference-attribute gap (MEDIUM, compile). No MIN_ALIGNMENT_*_BYTES prefs in hipBLASLt 7.2.1 -- guard out prod.cpp:620-623. Also verify CUBLAS_COMPUTE_32F_FAST_16F mapping and that hipblaslt heuristic returns >=1 algo for Marian's shapes (else fall back to a hipblasGemmEx + manual bias/ReLU epilogue, which Marian already has as the non-Lt path).
5. Legacy cuda_add_library (MEDIUM, build). Needs the cupoch shim macro; mark BOTH .cu and the mixed .cpp (prod.cpp, prod_sparse.cpp) LANGUAGE HIP since they include cublasLt/cusparse and template device code. Do not link roc::rocthrust/hip::device PUBLIC (host .cpp would get --offload-arch).
6. thrust::cuda::par -> thrust::hip::par (LOW, compile) cumsum.cu:62,110.
7. NCCL submodule unbuildable under hipcc (LOW, scoped out via USE_NCCL=OFF for lead; RCCL is a follow-up).
8. --use_fast_math + -ffp-contract default (LOW, numeric). clang/HIP defaults to -ffp-contract=fast; pin `-ffp-contract=on` to keep float math close to CUDA for any tolerance-sensitive unit test (CV-CUDA lesson). Marian unit tests use approximate tolerances so this is low-risk but cheap to set.
9. No textures/surfaces/managed memory -> those entire fault classes are N/A (positive finding).

## Build commands (gfx90a)
Prereqs: ROCm 7.2.1 at /opt/rocm; build with USE_CUDNN=OFF, USE_NCCL=OFF for the lead. Init submodules needed for the build (sentencepiece, fbgemm if USE_FBGEMM, NOT nccl). Build in a scratch dir under agent_space/ or the fork build/, never the MOAT repo root.
```
git submodule update --init src/3rd_party/sentencepiece
cmake -S . -B build-hip -G Ninja \
  -DUSE_HIP=ON -DCMAKE_HIP_ARCHITECTURES=gfx90a \
  -DCMAKE_HIP_COMPILER=/opt/rocm/llvm/bin/clang++ \
  -DCOMPILE_CUDA=ON -DUSE_CUDNN=OFF -DUSE_NCCL=OFF \
  -DUSE_FBGEMM=OFF -DCOMPILE_CPU=ON -DCMAKE_BUILD_TYPE=Release \
  -DBUILD_ARCH=native
cmake --build build-hip -j
```
(COMPILE_CUDA stays ON because it is the "build the GPU backend" switch; USE_HIP retargets it. Follower arches: only change -DCMAKE_HIP_ARCHITECTURES=gfx1100/gfx1151, no source edit.)

## Test plan (real GPU gate + non-GPU regression set)
- PRIMARY GPU gate (real gfx90a): the Catch2 unit suite in src/tests/units, which runs on DeviceType::gpu device 0. Targets run_operator_tests, run_attention_tests, run_transformer_tests, run_rnn_tests, run_graph_tests (operator_tests.cpp:1247 `tests<float>(DeviceType::gpu)`; the fp16 path is gated on compute capability via getCudaComputeCapability). These exercise element-wise ops, softmax/layernorm, reductions (add_all/reduce_all.h), prod/affine (hipBLAS + hipBLASLt epilogue), attention and the full Transformer forward/backward on GPU. Run via ctest SERIALLY on one GPU (`ctest -R 'operator|attention|transformer|rnn|graph'`, NOT -jN -- faiss/MPPI single-GPU contention lesson).
- SECONDARY GPU gate (end-to-end, the topk/nth_element + BLEU path): a small train+translate run. Use a tiny model from regression-tests (submodule marian-nmt/marian-regression-tests) or examples; train a few hundred steps on a toy parallel corpus and decode, then either (a) compare decoder output to the same build's CPU decode (`--cpu-threads`) for token agreement, or (b) decode a published small model and check BLEU against the reference within tolerance. This is the decisive test for the wave64 nth_element beam-search fix -- a corrupted argmax shows as garbage translations / BLEU collapse while unit tests still pass.
- DETERMINISM: run the topk/nth_element-exercising decode twice with a fixed seed and assert bit-identical output (proves the wave64 reduction fix removed the race).
- NON-GPU REGRESSION (must not regress): the CPU-side app tests (src/tests: test_logger, test_dropout, test_sqlite, test_cli) and the CPU branch of the unit tests (operator_tests also runs DeviceType::cpu cases). These are pure host C++, untouched by the HIP work (the compat header and LANGUAGE-HIP only touch .cu/GPU .cpp), so they must stay green. Build a CUDA-path or CPU-only config has no source delta.
- A CPU-only docker compile (rocm/dev-ubuntu-24.04:7.2.4-complete) is a fine manual compile smoketest but is NOT the gate (no GPU = cannot catch the wave64 reduction bug).

## Disposition
PROCEED -- TRACTABLE, MEDIUM-LARGE effort. The feared cuDNN/MIOpen wall does NOT exist: cuDNN is OFF by default and fully #ifdef-isolated, so MIOpen is unnecessary for the core NMT path. All required libraries (hipBLAS, hipBLASLt, hipSPARSE, hipRAND, rocThrust) are present in ROCm 7.2.1. No CUTLASS/tensor-core code; no textures/surfaces/managed memory. The custom kernel surface is small (16 TUs, one large element-wise file with no warp ops) and the only genuine wave64 correctness risk is the warp-synchronous reduction tail in topk.cu/nth_element.cu (a known, mechanically-fixable fault class). Main effort sinks: (1) the cublasLt->hipBLASLt affine epilogue port incl. the missing alignment-preference attrs, (2) the __CUDACC__/FP16 compat steering across types.h/operators.h, (3) the legacy cuda_add_library CMake shim. Scope cuDNN/MIOpen, NCCL/RCCL, and the sparse-attention cuSPARSE path as separable follow-ups (the default dense Transformer needs none of them). Not blocked.
