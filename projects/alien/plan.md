# alien -- ROCm/HIP port plan (lead platform: linux-gfx90a)

## Project
- Name: alien (Artificial Life Environment)
- Upstream: https://github.com/chrxh/alien (`chrxh/alien`)
- Default branch: `develop`
- What it is: a CUDA 2D particle-engine artificial-life / soft-body / fluid simulator. Simulation runs entirely on GPU; an OpenGL GUI renders via CUDA-OpenGL interop. ~28 `.cu` + ~75 `.cuh`, ~150 `__global__` kernels, all in `source/EngineKernels` and `source/EngineImpl`.

## Existing AMD support
None. README requires "an NVIDIA graphics card with compute capability 6.0 or higher"; the only GPU backend is CUDA; rendering is CUDA-OpenGL interop. No HIP/ROCm path, no OpenCL/Vulkan compute path. Decision: this is a genuine fresh CUDA->HIP port and a worthwhile MOAT target. PROCEED.

## Build classification: cmake (Strategy A)
Evidence:
- Top-level `CMakeLists.txt:29` `project(alien-project LANGUAGES C CXX CUDA)`; `.cu` sources added directly to `add_library(EngineKernels ...)` and `add_library(EngineImpl ...)`. No `find_package(Torch)`, no `torch.utils.cpp_extension`, no `CUDAExtension`, no setup.py/pyproject. Pure standalone CMake.
- `CMAKE_CUDA_ARCHITECTURES` defaults to `75` (`CMakeLists.txt:14-16`); `CMAKE_CUDA_SEPARABLE_COMPILATION ON` (`:21`) -- separable device compilation is used (kernels span TUs, `__constant__` extern symbols in `ConstantMemory.cuh`), so the HIP build needs `-fgpu-rdc` + `HIP_SEPARABLE_COMPILATION ON` (PORTING_GUIDE: RXMesh relocatable-device-code lesson).
- Third-party libs are vendored via a vcpkg submodule + manifest (`vcpkg.json`): glew, glfw3, glad, stb, boost-*, cereal, gtest, zlib, zstr, openssl, cli11, imgui (+freetype/glfw/opengl3 bindings), implot. All host-only; none are CUDA libraries; none need a HIP variant.

ext_type set to `cmake` in upstream.json and status.json.

## Port strategy: A (compat header + LANGUAGE HIP), colmap/MPPI model
Rationale: pure CMake, host C++ untouched, keep the NVIDIA path byte-for-byte. The CUDA surface is unusually clean (no cuBLAS/cuFFT/cuRAND/cuSPARSE, no Thrust/CUB, no textures/surfaces, no warp intrinsics), so the bulk is mechanical symbol aliasing plus a small number of real semantic fixes (graphs, cooperative-groups reduce, helper_cuda.h, GL interop, device selection).

Concrete shape:
1. Add `option(USE_HIP "Build with HIP for AMD GPUs" OFF)` at top-level. Under `USE_HIP`: `enable_language(HIP)`; default `CMAKE_HIP_ARCHITECTURES` to `gfx90a` ONLY when unset (never a literal in the target -- PORTING_GUIDE: configurable-arch lesson, CudaSift/Gpufit), so gfx1100/gfx1151 build with only `-DCMAKE_HIP_ARCHITECTURES=<arch>` and no source change. Else keep `enable_language(CUDA)` and the existing CUDA flags. Gate the NVCC-only flags (`-lineinfo --use-local-env -use_fast_math --threads 0 --Werror=all-warnings`) behind the CUDA branch; supply HIP equivalents (`-ffast-math`, drop `-lineinfo`).
2. Mark every `.cu` in EngineKernels/EngineImpl (+ the test/cli `.cu`) `LANGUAGE HIP` under `USE_HIP`. Because many per-target CMakeLists add `.cu` explicitly (not globbed), either set the property per target or use the MPPI top-level `add_library`/`add_executable` override that retags `.cu` to HIP + sets HIP_ARCHITECTURES + HIP_SEPARABLE_COMPILATION (zero churn in subdir CMakeLists). Apply `-fgpu-rdc`/`HIP_SEPARABLE_COMPILATION ON` to EngineKernels, EngineImpl and every consumer that links them (cli, EngineTests) -- the CUDA path already uses `CUDA_SEPARABLE_COMPILATION`/`CUDA_RESOLVE_DEVICE_SYMBOLS` for the same reason.
3. One forwarding shim dir on the HIP include path only (PORTING_GUIDE: MPPI shim-headers lesson) to satisfy the toolkit angle-bracket includes the sources use: `<cuda_runtime.h>`, `<device_launch_parameters.h>`, `<cuda_gl_interop.h>`, `<cooperative_groups.h>`, `<cooperative_groups/reduce.h>`, and `<cuda/helper_cuda.h>` / `<cuda/helper_string.h>` (the project's own `external/cuda/*`). The compat header (`cuda_to_hip.h`, force-included on HIP TUs via `CMAKE_HIP_FLAGS -include`) includes `<hip/hip_runtime.h>` (after `<cstring>/<cstdlib>` -- gpuRIR host-memcpy lesson) and aliases only the `cuda*` symbols actually used (enumerate from the grep surface below; use pytorch hipify `cuda_to_hip_mappings.py` as the authoritative name source).
4. Link: the CUDA path links `cudart`; on HIP link `hip::host` only (NOT `hip::device` -- cupoch/STRUMPACK `--offload-arch` propagation lesson). Map the bare `cudart` link to nothing-needed under HIP (the HIP runtime comes from the LANGUAGE-HIP toolchain + `hip::host`). `${CMAKE_CUDA_TOOLKIT_INCLUDE_DIRECTORIES}` PRIVATE includes become the HIP/compat include dir.

## CUDA surface inventory
- Kernels: ~150 `__global__` in `source/EngineKernels/*.cu`; orchestrated from `source/EngineImpl/SimulationKernelsService.cu` (`launchTimestepKernels`). Launch config via `KERNEL_CALL*` / `STREAM_KERNEL_CALL*` macros in `Macros.cuh` (default block size 8; explicit MODs of 4/16/32/64/NEURONS_PER_CELL/MAX_CHANNELS). Hand-rolled containers (`Array.cuh`, `HashMap.cuh`, `HashSet.cuh`, `List.cuh`, `Map.cuh`).
- Warp intrinsics: NONE. grep for `__shfl* / __ballot / __popc / warpSize / __activemask / __syncwarp / __any / __all / __lane*` across `source/` is EMPTY. No hardcoded-32 warp arithmetic in kernels (the only `<< 32` is a 64-bit key pack in `EditKernels.cu:505`). This eliminates the dominant wave64 fault class.
- Cooperative groups: `source/EngineKernels/ObjectProcessor.cuh` includes `<cooperative_groups.h>` + `<cooperative_groups/reduce.h>`, uses `cg::this_thread_block()`, `cg::tiled_partition<32>(block)`, and `cg::reduce(warp, x, cg::plus<float>())` (4 calls, SPH fluid force reduction in `calcFluidForces_reconnectCells_correctOverlap`). This is the ONE warp-cooperative construct in the codebase. Launched by `cudaNextTimestep_physics_calcFluidForces` via `STREAM_KERNEL_CALL` => block size 8 (tile<32> over an 8-thread block; CUDA handles the partial tile).
- Libraries: NO cuBLAS / cuFFT / cuRAND / cuSPARSE / cuSOLVER / Thrust / CUB. RNG is hand-rolled (`CudaNumberGenerator.cuh`: a pre-filled int array indexed by `atomicInc`).
- Textures/surfaces: NONE in device code (the "texture" hits are all OpenGL textures in `source/Gui`). No `cudaArray`, `cudaTextureObject`, `surf*`, `tex*`. The 256B-pitch / linear-filter / layered-array fault classes DO NOT APPLY.
- CUDA-OpenGL interop: `source/EngineKernels/CudaGeometryBuffers.cu` (`#include <cuda_gl_interop.h>`, `cudaGraphicsGLRegisterBuffer` with `cudaGraphicsMapFlagsWriteDiscard`) + `source/EngineImpl/GeometryKernelsService.cu` (`cudaGraphicsMapResources` / `cudaGraphicsResourceGetMappedPointer` / `cudaGraphicsUnmapResources` for 9 VBOs). `cudaGraphicsResource*` stored in `SimulationCudaFacade.cuh` and `CudaGeometryBuffers.cuh`. HIP equivalents exist 1:1 (`hipGraphicsGLRegisterBuffer`, `hipGraphicsMapResources`, `hipGraphicsResourceGetMappedPointer`, `hipGraphicsUnmapResources`, `hipGraphicsMapFlagsWriteDiscard`). Only reached on the RENDER path (GUI), NOT from `calcTimesteps`/`calcTimestep` -- so headless compute does not exercise it.
- CUDA Graphs: `source/EngineImpl/SimulationKernelsService.cu` -- `cudaStreamBeginCapture(cudaStreamCaptureModeGlobal)` / `cudaStreamEndCapture` / `cudaGraphInstantiate` / `cudaGraphLaunch` / `cudaGraphExecDestroy` / `cudaGraphDestroy`, per-config graph cache. This is the timestep execution path in non-debug mode. HIP equivalents exist (`hipStreamBeginCapture`, `hipGraph_t`, `hipGraphExec_t`, `hipGraphInstantiate(exec,graph,nullptr,nullptr,0)`, `hipGraphLaunch`). NOTE: `cudaGraphInstantiate` 5-arg signature -- the HIP overload is `hipGraphInstantiate(exec, graph, nullptr, nullptr, 0)` (matches). KEY DE-RISK: debug mode (`GlobalSettings::isDebugMode()`, set by `-d`) bypasses graphs entirely and direct-launches every kernel (`SimulationKernelsService.cu:191`).
- Atomics: atomicAdd (112), atomicExch (26), atomicMax (5), atomicMin (4), atomicCAS (4), atomicSub (3), atomicOr (3), atomicInc (1). Helpers in `Base.cuh` reinterpret to `unsigned long long`/`unsigned int`/`int`. `alienAtomicMax(float*)` uses signbit-split int/uint atomicMax/atomicMin (`Base.cuh:192-193`). All on `cudaMalloc` DEVICE memory (see below).
- Memory: plain `cudaMalloc`/`cudaFree` via `CudaMemoryManager` singleton; `cudaMemcpy`/`cudaMemset`. NO `cudaMallocManaged`, NO `cudaMallocHost`/pinned, NO `cudaMemAdvise`/prefetch. => the cudaKDTree "int atomicMin/Max silently dropped on coarse-grained/managed memory" fault class DOES NOT APPLY (that bug is managed-memory-specific; on `hipMalloc` device memory int/uint atomicMin/Max work).
- Streams/events: 1 `cudaStreamCreate`, `cudaStreamSynchronize` (x7), `cudaDeviceSynchronize` (x34). Modest; 1:1 HIP map.
- `__constant__` memory: `cudaSimulationParameters`, `cudaSettings` (`ConstantMemory.cu/.cuh`, `extern` across TUs -- another reason rdc is required). 1:1 HIP map (`__constant__` + `hipMemcpyToSymbol`).
- Device selection: `SimulationCudaFacade.cu:701-718` picks the device with the highest `prop.major*100+prop.minor` and rejects `<600`. On AMD, `hipDeviceProp_t.major/minor` is the gfx arch (e.g. gfx90a reports major 9), so the `< 600` gate would REJECT every AMD GPU. Must be guarded on HIP (accept all, or gate differently).
- Vendored `external/cuda/helper_cuda.h` + `helper_string.h`: included widely (`Macros.cuh`, `Array.cuh`, `Definitions.cuh`, `CudaMemoryManager.cuh`, `CudaNumberGenerator.cuh`, `SimulationCudaFacade.cu`). Only `checkCudaErrors` is actually CALLED; but the whole header must PARSE under hipcc. It contains a large `_cudaGetErrorEnum`/`cudaError*` switch and an NVIDIA SM-version-to-cores table + `_ConvertSMVer2Cores`/`gpuGetMaxGflopsDeviceId`/`gpuDeviceInit` (uses `cudaDevAttrComputeMode`, driver-ish bits). Cleanest fix: forward `<cuda/helper_cuda.h>` on HIP to a tiny shim providing just `checkCudaErrors` (+ `getLastCudaError` if used) over `hipError_t`, rather than trying to hipify the upstream helper wholesale. Keep the original on CUDA.

## Risk list (ranked)
1. CUDA Graphs / stream capture on ROCm 7.2.1 (HIGH). The hot path captures the full ~40-kernel timestep into a graph and replays it. HIP stream capture + hipGraph exist but are less battle-tested; capture-mode restrictions and any disallowed call inside capture (e.g. a sync, an alloc, or a `cudaGetLastError`) can fail capture on HIP. MITIGATION: built-in non-graph path via debug mode (`-d`) -- validate first with `-d` (direct launch) to prove kernel correctness independent of graphs, then enable graphs. If HIP graph capture is unreliable, gate the graph path off on HIP (force `launchTimestepKernels` + `hipStreamSynchronize`) -- a correctness-first mechanical port; note perf follow-up.
2. Cooperative-groups `cg::reduce` (HIGH/known). HIP CG (ROCm 7.2.1) has NO `cg::reduce` and no `cg::plus/greater/less` (PORTING_GUIDE: gsplat, cuCollections). The 4 `cg::reduce(warp, v, cg::plus<float>())` in `ObjectProcessor.cuh` must be replaced (USE_HIP/USE_ROCM-guarded) with a shfl_xor butterfly over the tile: `for(int o=warp.size()/2;o>0;o>>=1) v+=warp.shfl_xor(v,o)`. `tiled_partition<32>` is tile-relative-correct on wave64 (cuCollections lesson), and here the block is only 8 threads so `warp.size()` is the active span -- the butterfly bounded by `warp.size()` is correct on both wave32 and wave64. Provide a `<cooperative_groups/reduce.h>` shim header (cuCollections model) so call sites are nearly untouched. Also: `<cooperative_groups.h>`+`reduce.h` map to `<hip/hip_cooperative_groups.h>`.
3. CUDA-OpenGL interop under ROCm (MEDIUM, render-path only). `hipGraphicsGLRegisterBuffer` & friends exist but interop correctness on the gfx90a headless host is unproven and needs a GL context. Headless compute (CLI, EngineTests) does NOT hit it, so it does not gate GPU correctness validation. Port it mechanically (symbol aliases), defer interop runtime proof to the GUI path; if interop is flaky on ROCm, it is a GUI-only limitation, not a sim-correctness blocker.
4. `helper_cuda.h` parse under hipcc (MEDIUM). NVIDIA SM tables + driver attrs. Fix by shimming the header on HIP to just the used `checkCudaErrors` (see inventory). Watch for `cudaError` enum `case` labels with no HIP equivalent if any are referenced (MPPI/raft orphan-case lesson) -- the project's own switch in `Macros.cuh` references only common codes (InsufficientDriver, OperatingSystem, InitializationError, UnsupportedPtxVersion, MemoryAllocation, IllegalAddress, LaunchFailure) which all exist in HIP.
5. Device-selection `< 600` compute-capability gate (MEDIUM, certain). `SimulationCudaFacade.cu:717` throws on AMD (major 9 -> 900 passes, but gfx1100 major 11 -> 1100 passes; HOWEVER major/minor semantics differ and the heuristic `major*100+minor` is meaningless on AMD). Guard the gate and the "highest CC" selection on HIP (select device 0 or by free memory; accept any).
6. `-Werror` / `--Werror=all-warnings` (MEDIUM). The build treats all CXX and CUDA warnings as errors (`CMakeLists.txt:47,52`). clang/hipcc emits different warnings than nvcc/gcc (e.g. unused-result on nodiscard HIP event/timing calls -- amgcl lesson; `_Float16`/unused vars). Expect to relax `-Werror` on the HIP/clang compile or fix warnings; do NOT silence the NVIDIA path.
7. C++23 host / C++20 CUDA standard (LOW/MEDIUM). `CMAKE_CXX_STANDARD 23`, `CMAKE_CUDA_STANDARD 20`. hipcc/clang must accept `-std=c++20` for HIP TUs (it does on ROCm 7.2). If any rocPRIM/hipCUB header were pulled it would need C++17+ (GPUMD lesson) -- but alien pulls none, so this is just confirming clang accepts the chosen std for the `.cu`.
8. atomic float min/max via int/uint reinterpret (LOW). `alienAtomicMax(float*)` works on device memory on CDNA; flagged only because the cudaKDTree managed-memory bug is in the same family -- it does NOT apply here (device memory, not managed). Verify with the determinism/physics checks below rather than pre-emptively rewriting.
9. Two-phase-lookup / `__CUDA_ARCH__`-undefined-on-HIP family (LOW but check). grep shows no `#ifdef __CUDA_ARCH__` device/host data-pointer selection in the kernel headers, and `__host__ __device__` is used directly (not via a `__CUDACC__`-gated macro). Spot-check during bringup for any `this->STATIC` non-type-template-arg or dependent-base unqualified calls (MPPI/cudaKDTree) -- low likelihood given the code style.
10. NVCC-only flags (`-lineinfo`, `--use-local-env`, `-use_fast_math`, `--threads`, `--Werror=all-warnings`) on the HIP compile (LOW, build-config). Gate them to the CUDA branch.

NOT applicable (ruled out by grep): warpSize 32-vs-64 in kernels, `__shfl/__ballot` masks, textures/surfaces (256B pitch, linear filtering, layered arrays), cuBLAS/cuFFT/cuRAND/cuSPARSE swaps, Thrust/CUB, managed-memory atomic drop, pinned-memory.

## Build commands (gfx90a)
Dependencies via the project's vcpkg submodule (host libs; install OS X11/GL dev packages as the CI does: `libx11-dev libxcursor-dev libxrandr-dev libxinerama-dev libxi-dev libxext-dev libxfixes-dev libgl1-mesa-dev libglu-dev`). Bootstrap vcpkg (`external/vcpkg/bootstrap-vcpkg.sh -disableMetrics`).

Configure + build (lead arch gfx90a):
```
cmake -S projects/alien/src -B projects/alien/src/build \
  -DCMAKE_TOOLCHAIN_FILE=projects/alien/src/external/vcpkg/scripts/buildsystems/vcpkg.cmake \
  -DCMAKE_BUILD_TYPE=Release \
  -DUSE_HIP=ON -DCMAKE_HIP_ARCHITECTURES=gfx90a \
  -DCMAKE_HIP_COMPILER=/opt/rocm/llvm/bin/clang++
cmake --build projects/alien/src/build -j32
```
(Same command builds gfx1100/gfx1151 with only `-DCMAKE_HIP_ARCHITECTURES=<arch>`.) A CPU-only compile smoketest can use `rocm/dev-ubuntu-24.04:7.2.4-complete` locally (never wired into Actions; never a validation gate).
After `gh repo fork`, disable Actions on the jeffdaily fork.

## Test plan
Real GPU suites (gtest, built by the same CMake):
- `EngineTests` -- the GPU simulation suite (39 files: Attacker, Constructor, Muscle, Neuron, Physics, Mutation, Sensor, Detonator, Digestor, Injector, Reconnector, Communicator, Generator, Defender, Depot, EnergyFlow, EnergyParticle, CellStateTransition, Creature, GarbageCollector, Edit, DescriptionEdit, DataTransfer, HeadUpdate, LayerParameter, Memory, ObjectConnection, Radiation, SimulationControl, Statistics, Void, Geometry). Each builds `_SimulationFacadeImpl` and runs real timesteps on GPU asserting on resulting state. Run binary directly: `cd build && ./EngineTests`. Use `./EngineTests -d` FIRST (debug mode = direct kernel launch, bypasses CUDA graphs) to validate kernel correctness in isolation; then run without `-d` to validate the graph path. Run SERIALLY on a single assigned GPU (no parallel gtest processes -- MPPI single-GPU contention lesson).
  - GL caveat: `GeometryTests.cpp` (and only it) calls `glfwInit()/glfwCreateWindow(100,100,hidden)/gladLoadGLLoader` to exercise the CUDA-GL interop geometry buffers. On a headless validator without a display this suite may fail at context creation -- run under an offscreen GL (EGL / xvfb-run / `glfw` with `GLFW_VISIBLE` off) or use a gtest filter to scope it out and document it; it does NOT gate the non-GL GPU correctness which is the bulk of the suite.
- Headless CLI smoke (no GL context needed): `./cli -i <example.sim> -o /tmp/out.sim -t 1000` runs 1000 timesteps on GPU and writes the resulting `.sim` + `.statistics.csv`. Use a bundled/representative example simulation as input. This is the simplest "kernels actually run on the device" smoke and exercises the full timestep pipeline incl. graph capture.

Determinism / correctness bar (chaotic many-body sim -- GPUMD/MPPI lesson): the simulation uses atomicAdd-ordered accumulation and a pre-seeded RNG, so bitwise run-to-run identity is NOT guaranteed and is not the bar. Validate by (a) the EngineTests assertions passing (these encode physics/biology invariants: energy flow, construction correctness, mutation behavior, attack/defense, connection forces), and (b) a CLI run completing N steps without device fault and producing a plausible statistics trajectory (object/energy counts conserved per the engine's rules). Confirm kernels execute on-device via `AMD_LOG_LEVEL=3` (named `cudaNextTimestep_*` dispatches).

Non-GPU regression set (must not regress; CI runs these): `EngineInterfaceTests`, `NetworkTests`, `PersisterTests` (and `EngineTests` non-GPU-ish data tests). `NetworkTests` hits an external server API -- may need network; treat network-dependent failures as environmental, not port regressions.

## Inter-project MOAT deps
None. All third-party deps are host-only and vcpkg-managed; alien does not consume any other MOAT-ported library (no rmm/raft/etc.). No `set-deps` needed.
